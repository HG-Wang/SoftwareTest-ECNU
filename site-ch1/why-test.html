为什么要测试

Why Do We Test Software — 软件故障的危害、故障/错误/失败的概念与区别、RIPR故障模型。

故障、错误与失败

在软件测试领域，Fault、Error和Failure是三个密切相关但含义不同的概念：

🐛 Software Fault（软件故障）

静态缺陷 — 软件中的一个静态存在的缺陷，如代码中的一个错误语句。在不执行程序的情况下就存在。

  • 也称为 Bug 或 Defect
  • 是代码中的一个"坏点"
  • 不执行就不会被触发

⚠️ Software Error（软件错误）

不正确的内部状态 — 故障被触发后，在程序内部产生的错误中间状态。是故障在执行中的表现。

  • 程序内部变量值不正确
  • 是故障和失败之间的桥梁
  • 外部通常不可直接观察

💥 Software Failure（软件失败）

外部错误行为 — 程序相对于需求规约的不正确的外部行为。是最终可观察到的错误输出。

  • 可被用户或测试人员观察到
  • 是错误传播到输出的结果
  • 可能影响系统可用性

医生诊断类比

类比理解： 将软件比作病人，Failure（失败）= 症状（如发烧），Fault（故障）= 病因（如细菌感染），Error（错误）= 异常内部状态（如白细胞升高）。医生需要通过症状(Failure)追溯病因(Fault)。

numZero 示例

以下代码段的目标是计算数组中值为0的元素个数，但存在一个故障：

// 有故障的代码
public static int numZero(int[] arr) {
    int count = 0;
    for (int i = 1; i < arr.length; i++) {  // ⚠️ Fault: 应该是 i = 0
        if (arr[i] == 0) {
            count++;
        }
    }
    return count;
}Java

Fault → Error → Failure 因果链

层次 描述 示例
Fault 循环起始条件 i=1 应为 i=0 代码第3行的 i=1
Error 数组第一个元素 arr[0] 没有被检查 count 可能比实际值少1
Failure 当 arr[0]=0 时，返回值比正确值少1 numZero({0,1,0}) 返回 1 而非 2
注意： Fault 不一定导致 Failure。如果 arr[0] ≠ 0，即使存在 i=1 的故障，也不会产生错误和失败。这体现了测试的困难性——故障可能隐藏在未被覆盖的代码路径中。

RIPR故障模型

一个软件故障要导致可观察的失败，需要满足四个条件：

R
Reachability
可达性
→
I
Infection
感染性
→
P
Propagation
传播性
→
R
Reveal
可观察性
R — Reachability（可达性）
包含故障的代码必须被执行到。如果程序的执行路径不经过故障位置，故障永远不会被触发。
I — Infection（感染性）
执行故障代码后，程序的内部状态必须被改变为不正确的状态。即故障必须产生一个错误(Error)。
P — Propagation（传播性）
不正确的内部状态必须传播到程序的输出，使得外部可观察到错误的行为。即错误必须导致失败(Failure)。
R — Reveal（可观察性）
测试人员或用户必须能够观察到错误的输出。即失败必须是可被检测到的。
关键洞察： RIPR模型解释了为什么测试是困难的——即使存在故障，也需要构造合适的输入使得四个条件同时满足，才能观察到失败。这也解释了为什么"测试显示缺陷存在，而非不存在"。

软件可测试性

软件可测试性(Testability)衡量发现软件故障的难易程度，由两个因素决定：

👁️ Observability（可观察性）

观察程序行为的容易程度。

  • 输出是否易于理解和验证
  • 中间状态是否可以被监控
  • 错误是否容易被发现

🎮 Controllability（可控制性）

提供所需输入的容易程度。

  • 是否容易构造特定的输入
  • 程序状态是否容易设置
  • 故障路径是否容易被触发

测试用例的组成

一个完整的测试用例包含以下组成部分：

组成部分 说明
Test Case Values（测试用例值） 直接驱动被测软件的输入值
Prefix Values（前缀值） 将软件置于正确初始状态所需的输入
Postfix Values（后缀值） 测试后清理或恢复状态所需的输入
Expected Results（期望结果） 测试执行后期望的输出或行为
Verification Values（验证值） 用于检查中间状态的值
Exit Values（退出值） 测试结束时系统的期望状态

验证与确认

Verification（验证）和Validation（确认）是软件质量保证中的两个重要概念：

🔍 Verification（验证）

"Are we building the product right?" — 检查开发过程产物是否满足前一阶段的需求。

  • 关注过程和中间产物
  • 通常是静态的（如评审、检查）
  • 在开发过程中持续进行

✅ Validation（确认）

"Are we building the right product?" — 评估软件是否符合用户的预期用途。

  • 关注最终产品和用户需求
  • 通常是动态的（如运行测试）
  • 主要在开发后期进行

案例：哈勃望远镜

哈勃望远镜案例： 哈勃太空望远镜的主镜片按照规格书精确制造（Verification 成功），但规格书本身存在错误——镜片的曲率设计有误（Validation 失败）。结果望远镜发射后图像模糊，需要花费数亿美元进行修复。这说明即使验证通过，也不能保证产品是正确的。

测试成熟度等级

软件测试的成熟度可以分为五个等级：

Level 0 — 测试 = 调试

没有区分测试和调试。测试的目的仅仅是证明软件能工作。没有正式的测试过程，开发者随意测试自己的代码。

Level 1 — 目的是证明正确性

测试的目的是证明软件是正确的。测试被视为开发的最后阶段，用于"显示软件工作正常"。这种观点的问题在于它暗示测试可以证明没有缺陷，而这是不可能的。

Level 2 — 目的是证明软件不工作

测试的目的是发现缺陷，证明软件不工作。这是一个重要的思维转变——从"证明正确"到"发现错误"。测试人员开始有意识地寻找软件中的问题。

Level 3 — 目的是降低使用风险

测试的目的是降低软件在使用中出现故障的风险。测试不再是简单的"通过/不通过"，而是为决策提供信息。测试成为一种风险评估活动。

Level 4 — 测试是提升质量的思维纪律

测试是一种预防性活动，是提升软件质量的思维纪律。测试活动贯穿整个开发生命周期，从需求阶段就开始参与。测试与开发深度融合。
