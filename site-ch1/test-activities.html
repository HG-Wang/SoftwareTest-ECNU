测试活动与模型驱动设计

Test Activities & Model-Driven Test Design — 软件测试的四大活动分类、模型驱动测试设计方法、测试自动化框架。

测试活动 MDTD 测试自动化

软件测试的复杂性

没有其他工程领域构建的产品像软件这样复杂。软件的复杂性体现在多个方面：

  • 输入空间巨大，穷举测试不可行
  • 软件行为依赖于上下文和环境
  • "正确性"对软件来说没有严格的数学定义——只能相对于需求规约来判断
  • 需求本身可能是不完整、不一致或错误的
核心思想： 需要使用抽象（Abstraction）来管理复杂性。通过将软件制品抽象为模型（Model），可以从模型中系统地推导出测试需求和测试用例——这就是模型驱动测试设计（MDTD）的核心思想。

四大测试活动

软件测试可以分解为四个主要活动，每个活动有不同的目标和技能要求：

Test Design（测试设计）

测试设计是四个活动中技术性最强的工作。测试设计师需要具备离散数学、编程和测试理论方面的深厚知识。

测试设计分为两种方法：

  • Criteria-based（基于标准的）： 使用覆盖标准来系统地生成测试用例，如本课程介绍的各种图覆盖标准和逻辑覆盖标准
  • Human-based（基于人工的）： 依靠测试人员的经验和直觉来设计测试用例，如探索性测试和基于经验的测试

Test Automation（测试自动化）

将测试设计阶段产出的测试值嵌入可执行的测试脚本中。自动化可以：

  • 减少成本： 回归测试可以反复自动执行
  • 减少人为错误： 避免手动执行中的疏忽
  • 提高效率： 可以在夜间或CI/CD流水线中自动运行

需要编写代码的能力和对测试框架的了解。

Test Execution（测试执行）

运行测试并记录结果。这是最直接的测试活动：

  • 执行自动化测试脚本或手动测试步骤
  • 记录测试结果（通过/失败/阻塞）
  • 记录实际输出和环境信息
  • 报告发现的问题

Test Evaluation（测试评估）

评估测试结果，向开发者报告发现的问题：

  • 分析测试结果，确定是否为真正的缺陷
  • 评估测试覆盖率和测试有效性
  • 编写缺陷报告，与开发者沟通
  • 需要对应用领域的深入理解

四种建模结构

建模结构 描述 典型来源
Input Domains（输入域/集合） 将输入空间划分为等价类或子集 输入参数的取值范围
Graphs（图） 用节点和边表示程序结构或行为 控制流图、状态机、调用图
Logical Expressions（逻辑表达式） 用逻辑公式表示条件和约束 谓词、条件分支、规约
Syntax Descriptions（语法描述/文法） 用文法规则描述输入的语法结构 协议格式、输入文件格式

果冻豆示例

假设一个果冻豆有6种口味（樱桃、柠檬、橙子、蓝莓、薄荷、巧克力）和4种颜色（红、黄、蓝、绿），共24种组合。

🍬 按口味覆盖

测试需求：每种口味至少尝一次。

最少需要 6个 测试用例。

满足口味覆盖不一定满足颜色覆盖（可能遗漏某些颜色）。

🎨 按颜色覆盖

测试需求：每种颜色至少尝一次。

最少需要 4个 测试用例。

满足颜色覆盖不一定满足口味覆盖（可能遗漏某些口味）。

包含关系： 口味覆盖不包含颜色覆盖，颜色覆盖不包含口味覆盖。这是两种独立的覆盖标准。但如果定义一个更强的标准"口味×颜色覆盖"（每种组合至少一次），则它同时包含口味覆盖和颜色覆盖。

测试自动化框架

现代测试自动化框架提供以下核心功能：

功能 说明 示例
断言（Assertions） 验证实际结果是否等于期望结果 JUnit的assertEquals, assert
共享测试数据 在多个测试之间共享设置和清理操作 setUp/tearDown, fixtures
测试集组织 将测试用例组织成套件和类别 TestSuite, test tags
命令行/GUI运行 支持多种方式运行测试 命令行、IDE插件、CI/CD集成

常用测试框架

  • JUnit — Java语言的单元测试框架
  • unittest — Python标准库的测试框架
  • Selenium — Web应用的UI自动化测试框架
  • pytest — Python的高级测试框架
  • Jest — JavaScript的测试框架

团队组织

测试团队通常按照以下角色进行组织：

角色 人数 技能要求
测试设计师 1人 需要CS背景，掌握离散数学和测试理论
测试自动化工程师 若干人 需要编程能力，熟悉测试框架和工具
测试执行人员 若干人 基本计算机技能，细心负责
测试评估人员 若干人 需要领域知识，能分析测试结果

Do and Don't — 测试最佳实践

✅ Do（应该做的）

  • 尽早开始测试
  • 使用模型驱动的系统化方法
  • 编写可重复执行的自动化测试
  • 记录和追踪缺陷
  • 关注高风险区域

❌ Don't（不应该做的）

  • 不要测试自己的代码（缺乏独立性）
  • 不要追求穷尽测试
  • 不要忽略回归测试
  • 不要在没有标准的情况下随意测试
  • 不要将测试推迟到开发结束后
