规约的逻辑覆盖

Logic Coverage for Specifications — 将逻辑覆盖标准应用于软件规约，利用 CNF/DNF 结构简化测试生成。

8.4 CNF DNF 前置条件

规约的形式与分类

规约（Specifications）可以是形式化的（用数学语言表达）或非形式化的（用自然语言表达）。

逻辑规约可在多个层级描述系统行为：

层级说明应用场景
方法/类级别描述单个方法或类的行为单元测试、模块测试
组件级别描述模块或组件间的交互集成测试
系统级别描述整个系统的全局行为系统测试

大多数规约语言包含显式的逻辑表达式，因此非常容易应用逻辑覆盖标准。自然语言规约中的隐式逻辑表达式应在测试设计时改写为显式逻辑表达式 — 你通常会在这一步发现错误。

前置条件示例

程序员经常为方法添加前置条件，通常以注释形式出现在方法头中：

// name must not be empty
// state must be valid
// zip must be 5 numeric digits
// street must not be empty
// city must not be empty
Comments

将上述前置条件改写为合取范式（CNF）的逻辑表达式：

name != ""
  ∧ state ∈ stateList
  ∧ zip ≥ 00000
  ∧ zip ≤ 99999
  ∧ street != ""
  ∧ city != ""
CNF

这是一个典型的 CNF 谓词 — 由多个子句通过 ∧（与） 连接。

CNF 快捷方法

合取范式 (CNF)： 谓词由子句或析取子句通过 ∧ 连接，如 A ∧ B ∧ C ∧ ... 或 (A ∨ B) ∧ (C ∨ D)。

对于 CNF 谓词，主子句在所有其他子句为 true 时被激活。

ACC 测试策略："全真"行 + 对角线上的假值组合：

测试ABC…说明
1 T T T … 所有子句为 true → 谓词为 true
2 F T T … A 为 false → A 决定谓词（false）
3 T F T … B 为 false → B 决定谓词（false）
4 T T F … C 为 false → C 决定谓词（false）
规律： 对 N 个子句的 CNF 谓词，只需 N + 1 个测试即可满足 ACC — 一个"全真"测试 + N 个"轮流假"测试。

DNF 快捷方法

析取范式 (DNF)： 谓词由子句或合取子句通过 ∨ 连接，如 A ∨ B ∨ C ∨ ... 或 (A ∧ B) ∨ (C ∧ D)。

对于 DNF 谓词，主子句在所有其他子句为 false 时被激活。

ACC 测试策略："全假"行 + 对角线上的真值组合：

测试ABC…说明
1 F F F … 所有子句为 false → 谓词为 false
2 T F F … A 为 true → A 决定谓词（true）
3 F T F … B 为 true → B 决定谓词（true）
4 F F T … C 为 true → C 决定谓词（true）
规律： 对 N 个子句的 DNF 谓词，同样只需 N + 1 个测试即可满足 ACC — 一个"全假"测试 + N 个"轮流真"测试。

CNF 与 DNF 快捷方法对比

🔗 CNF（合取范式）

  • 结构：A ∧ B ∧ C
  • 激活条件：其他子句全为 true
  • 测试模式：全真 + 轮流假
  • 每个测试使一个子句决定谓词
  • 适用于：前置条件、约束

🔀 DNF（析取范式）

  • 结构：A ∨ B ∨ C
  • 激活条件：其他子句全为 false
  • 测试模式：全假 + 轮流真
  • 每个测试使一个子句决定谓词
  • 适用于：触发条件、选项

逻辑规约的来源

逻辑规约可以从多种来源获得：

来源说明示例
前置条件 方法执行前必须满足的条件 // requires: list != null
Java Assert 代码中的断言检查 assert x > 0 : "x must be positive"
契约 Design-by-Contract 中的前置/后置条件和不变量 @pre: balance >= 0
OCL 条件 UML 模型中的对象约束语言 context Person inv: age >= 0
形式化语言 Z、VDM 等数学规约语言 数学化的状态机和不变量
实践价值： 规约中的许多谓词天然处于 DNF 或 CNF 形式，这大大简化了测试计算。由于规约中的谓词通常具有规则结构，可以直接使用 CNF/DNF 快捷方法计算 ACC 测试，无需逐一分析每个子句的活动条件。
