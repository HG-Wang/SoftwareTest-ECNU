源代码逻辑覆盖 (Logic Coverage for Source Code)

将逻辑覆盖标准应用于实际源代码时，需要处理可达性、可控性和内部变量等挑战。

§ 8.3 可达性分析 谓词提取

基本步骤#

  1. 提取谓词 (Extract Predicates)

    从源代码中的条件语句（if、while、for 等）中提取逻辑谓词。

  2. 分析可达性 (Analyze Reachability)

    确定到达包含谓词的代码语句所需的路径条件和前置约束。

  3. 求解内部变量 (Solve Internal Variables)

    处理谓词中出现的非输入变量，建立它们与输入变量的关系。

  4. 求解约束 (Solve Constraints)

    将覆盖标准产生的逻辑约束转化为可求解的输入值约束。

  5. 推导测试用例 (Derive Tests)

    从约束求解结果中生成具体的测试输入值。

核心挑战#

🎯

可达性 (Reachability)

如何到达包含目标谓词的代码语句？需要满足所有前置条件和路径约束。

🎮

可控性 (Controllability)

如何通过外部输入控制谓词中各子句的取值？输入变量与子句之间的映射关系可能很复杂。

🔗

内部变量 (Internal Variables)

谓词中可能包含非输入的中间变量，需要回溯计算其与输入的关系。

三角形分类示例#

从三角形分类程序中提取谓词的过程：

int classify(int s1, int s2, int s3) {
  int triOut;
  // 计算三角形类型
  if (s1 == s2 && s2 == s3)       triOut = 2;  // 等边
  else if (s1 != s2 && s2 != s3 && s1 != s3) triOut = 4;  // 不等边
  else                              triOut = 3;  // 等腰

  if (s1+s2 > s3 && s2+s3 > s1 && s1+s3 > s2) {
    return triOut;  // 有效三角形
  } else {
    return 0;      // 无效
  }
}

triOut 有 6 种可能取值 (0-5)，映射到不同的三角形类型。谓词提取后需分析每个条件分支的可达性约束。

恒温器示例#

考虑恒温器控制程序中的谓词：

if ((a || (b && c)) && d) {
  turnOnHeater();
}

其中各子句的定义：

子句含义输入变量
acurTemp < dTemp - thresholdDiff当前温度、期望温度、阈值差
bOverride用户覆盖开关
ccurTemp < overTemp - thresholdDiff当前温度、覆盖温度、阈值差
dtimeSinceLastRun > minLag上次运行时间、最小间隔

可达性分析

要到达该 if 语句，需要确保程序执行路径到达此处。可达性约束可能包括前置状态检查等。

CACC 测试推导

对于 CACC 覆盖，该谓词需要 6 个测试用例：

测试目标子句子句值谓词值Pa (其他约束)说明
T1aTT!(b&&c) && d 温度低于阈值，非覆盖模式，时间间隔足够
T2aFT(b&&c) && d 温度正常，但覆盖模式开启且冷却时间足够
T3bTT!a && c && d 覆盖开启，温度低于覆盖温度，间隔足够
T4bFTa && d 覆盖关闭，但温度已低于阈值
T5cTT!a && b && d 温度低于覆盖温度，覆盖开启
T6cFF!a && b && d 温度高于覆盖温度，覆盖开启但谓词为假

每个测试用例需要将逻辑约束转化为具体的温度、时间等输入值。

程序变换问题#

对程序进行变换后再应用逻辑覆盖标准，可能导致覆盖不充分的问题。

变换 1：嵌套 if 语句

原始 
if (A && B) {
  // ...
}
变换后 
if (A) {
  if (B) {
    // ...
  }
}

对变换后的代码执行 PC (Predicate Coverage)，不能保证对原始代码满足 CACC。因为嵌套 if 将一个复合谓词拆分为两个独立判定，丢失了子句间的关联信息。

在变换后的代码上满足 PC，需要 更多 的测试用例才能达到原始代码的 CACC 覆盖。

变换 2：引入中间变量

原始 
if (x > 0 && x < 100) {
  // ...
}
变换后 
bool a = (x > 0);
bool b = (x < 100);
if (a && b) {
  // ...
}

这种变换将复杂度从逻辑谓词转移到了计算过程中。虽然谓词变得简单（只有布尔变量），但逻辑覆盖标准无法有效覆盖原始的比较运算。

结论：程序变换 (transformation) 不可行，不应在变换后的代码上应用逻辑覆盖标准。

谓词中的副作用#

当谓词中包含具有副作用的函数调用时，逻辑覆盖分析变得更加复杂：

if (A && (B || changeVar(A))) {
  // ...
}

其中 changeVar(A) 不仅返回一个布尔值，还可能修改某些状态。这导致：

  • 子句 B 和 changeVar(A) 的求值顺序会影响结果
  • 短路求值 (short-circuit evaluation) 可能跳过副作用
  • 同一谓词在不同执行中可能产生不同结果
包含副作用的谓词使得逻辑覆盖标准的假设（谓词是纯布尔函数）不再成立，分析时需要特别小心。

总结#

要点说明
谓词规模 大多数实际程序的谓词 < 4 个子句，组合覆盖代价可接受
困难所在 内部变量的求解是最大的挑战，需要回溯数据流
避免变换 不要对程序进行变换后再应用逻辑覆盖标准
工具支持 使用约束求解器 (SMT solver) 可自动化测试用例生成
实践建议 优先使用 MC/DC，对关键判定使用组合覆盖
