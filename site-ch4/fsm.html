有限状态机的逻辑覆盖

Logic Coverage of FSMs — 将逻辑覆盖标准应用于有限状态机的转换守卫条件，系统地生成测试用例。

8.5 FSM 守卫条件 CACC

FSM 基本概念

有限状态机（Finite State Machine, FSM）本质上是图：

  • 节点（Nodes） = 状态（States）
  • 边（Edges） = 转换（Transitions），带有逻辑守卫或触发条件

FSM 广泛用于各层级的抽象建模，表达形式包括 Statecharts、状态表、Z 记法、决策表、Petri 网等。

核心思路： 找到转换上的逻辑表达式，然后覆盖它 — 这就是 FSM 逻辑覆盖的本质。

转换守卫条件

进入"Left Doors Open"状态的守卫条件：

trainSpeed = 0
  ∧ platform = left
  ∧ (location = inStation
      ∨ (emergencyStop ∧ overrideOpen ∧ location = inTunnel))
Guard

简化为子句形式（6 个子句）：

a ∧ b ∧ (c ∨ (d ∧ e ∧ f))
Simplified
子句变量含义
atrainSpeed = 0列车速度为零
bplatform = left站台在左侧
clocation = inStation位于站内
demergencyStop紧急停车
eoverrideOpen覆盖开启
flocation = inTunnel位于隧道内

判定分析（Determination）

使用 XOR 方法计算每个子句决定谓词的条件：

// 定义法：Pc = P_{c=true} ⊕ P_{c=false}
Pa = b ∧ (c ∨ (d ∧ e ∧ f))
Pb = a ∧ (c ∨ (d ∧ e ∧ f))
Pc = a ∧ b ∧ ¬(d ∧ e ∧ f)
Pd = a ∧ b ∧ ¬c ∧ e ∧ f
Pe = a ∧ b ∧ ¬c ∧ d ∧ f
Pf = a ∧ b ∧ ¬c ∧ d ∧ e
Determination
子句Pc 条件含义
ab ∧ (c ∨ (d∧e∧f))需要 b=true，且 c 或 (d∧e∧f) 为 true
ba ∧ (c ∨ (d∧e∧f))需要 a=true，且 c 或 (d∧e∧f) 为 true
ca ∧ b ∧ ¬(d∧e∧f)需要 a、b 为 true，且 d∧e∧f 为 false
da ∧ b ∧ ¬c ∧ e ∧ f需要 a、b、e、f 为 true，c 为 false
ea ∧ b ∧ ¬c ∧ d ∧ f需要 a、b、d、f 为 true，c 为 false
fa ∧ b ∧ ¬c ∧ d ∧ e需要 a、b、d、e 为 true，c 为 false

发现的问题与挑战

不可行测试需求

问题： inStation 和 inTunnel 不能同时为假！模型只有 2 个位置，因此 c = F ∧ f = F 是不可能的组合。
解决： 更改真值赋值（如改为 c=F, f=T），或与开发者确认模型约束。

预期结果与意外转换

预期结果从 FSM 中读取：

  • 守卫条件为 true → 转换被触发
  • 守卫条件为 false → 转换不被触发
意外转换： 某个转换的 false 结果可能恰好是另一个转换的 true。例如 platform ≠ left 时守卫为 false，但如果 platform = right，系统将转换到"Right Doors Open"状态，而非停留在原状态。测试自动化时必须识别这些意外转换。

测试自动化挑战

挑战说明示例
映射问题 FSM 中的名称与程序中的名称可能不一致 FSM 用 platform=left，程序中需要调用 goToStation("West")
可达性 测试必须先到达目标状态所在的转换起点 需要先执行一系列转换将列车开到站台

总结

  • FSM 被广泛使用，特别是在嵌入式软件中
  • 转换上的谓词是 FSM 逻辑覆盖的核心
  • 谓词通常以守卫条件、动作、安全约束的形式显式出现在转换上
  • FSM 有多种表达形式：Statecharts、状态表、Z 记法、决策表、Petri 网等
