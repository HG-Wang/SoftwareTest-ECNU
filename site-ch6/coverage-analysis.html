覆盖分析

Coverage Analysis — 验证测试是否触达关键路径，使用 gcov/lcov 统计覆盖率。

第四部分 gcov 语句覆盖 判定覆盖

核心理解：覆盖分析用于检验测试决策是否触达高风险路径，而不是驱动新增测试。

测试执行

使用自动化脚本实现并运行测试集合。

测试脚本模板

#!/bin/bash

# 清理旧文件
rm -f coreutils/echo coreutils/*.gcda coreutils/*.gcno coreutils/*.info
rm -rf coreutils/coverage_report/

# 编译（启用覆盖率统计）
cd /home/busybox/Desktop/busybox
gcc -Iinclude -Ilibbb -include include/autoconf.h \
    -fprofile-arcs -ftest-coverage -g \
    -c -o coreutils/echo.o coreutils/echo.c
gcc -o coreutils/echo coreutils/echo.o \
    -fprofile-arcs -ftest-coverage \
    -Wl,--start-group shell/lib.a libbb/lib.a -Wl,--end-group

# 运行测试用例
cd coreutils
./echo -e "hello\cworld"
./echo -n -e "a\nb"
./echo -e "a\tb"
./echo a b
./echo

# 收集覆盖率数据
lcov --capture --directory . \
    --output-file coverage.info \
    --rc lcov_branch_coverage=1

# 过滤覆盖率数据
lcov --remove coverage.info "/usr/*" "*/tests/*" \
    --output-file coverage_filtered.info \
    --rc lcov_branch_coverage=1

# 生成 HTML 报告
genhtml coverage_filtered.info \
    --output-directory coverage_report \
    --branch-coverage
约束：保证测试可重复执行；确保测试结果可复现；不得为提高覆盖率临时修改测试。
通用性：测试其他工具时，只需替换工具名和测试用例部分，其余结构完全通用。建议每次测试使用独立的测试目录。

覆盖结果分析

未覆盖分支主要包括以下三种场景：

未覆盖路径：当用户输入无效选项（如 -z）时，程序会将参数原样回显。

风险分析：不会崩溃或产生安全问题，只是功能降级，不是功能失效。

是否纳入原始测试集合：否 — 低风险、低频场景。

未覆盖路径：八进制转义序列（如 \0141）的解析逻辑。

风险分析：很少使用的特性，有默认行为，影响范围小。

是否纳入原始测试集合：否 — 低频使用特性。

未覆盖路径：写入错误（如磁盘满）时的错误处理逻辑。

风险分析：需要特殊环境才能触发，错误处理是防御性编程，正常环境不会执行。

是否纳入原始测试集合：否 — 需要特殊环境，低频触发。

结论：未覆盖分支均为低风险、低频、防御性代码，符合风险驱动的取舍原则。

关键结论

该示例说明了两个核心概念的区别：

代码覆盖率

回答：测试是否触达关键程序结构？

覆盖率是结构触达的度量，用于检验测试决策的有效性。

风险驱动测试

回答：测试是否控制关键行为风险？

风险驱动关注的是行为控制的有效性，而非结构覆盖的完整性。

关键结论：高覆盖率并不必然意味着高测试价值。提高覆盖率通常需要设计额外测试、增加测试执行成本和维护成本。在实际工程中，测试设计需要在风险控制与测试成本之间进行权衡。
