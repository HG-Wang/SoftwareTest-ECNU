3.5 FSM 顺序约束 状态机

顺序约束

什么是顺序约束？

顺序约束（Sequencing Constraints）是对方法调用顺序施加限制的规则。它们可以作为前置条件或其他规约来表达。类中的方法可能互不调用，但可以通过方法调用序列来创建测试，顺序约束提供了选择哪些序列的有效方式。

顺序约束的来源

  • 显式表达的约束
  • 隐式表达的约束
  • 完全没有表达的约束
  • 测试者应在约束不存在时推导它们：查看设计文档、需求文档，询问开发者（最后选择：查看实现）

文件ADT示例

class FileADT 有三个方法：open(fName), close(), write(textLine)

有效的顺序约束：

  1. 每次 write(t) 之前必须有 open(f)
  2. 每次 close() 之前必须有 open(f)
  3. close() 之后不能执行 write(t)，除非中间有 open(f)
  4. 每次 close() 之前应该有 write(t)

静态检查

通过分析调用图，检查是否存在违反顺序约束的路径：

  • 是否存在不经过 open() 就到达 write() 的路径？
  • 是否存在不经过 open() 就到达 close() 的路径？
  • 是否存在从 close() 到 write() 的路径？
  • 是否存在从 open() 到 close() 但不经过 write() 的路径？

测试需求生成

使用顺序约束生成测试需求，目标是违反每个顺序约束。如果程序正确，所有测试需求都应该是不可行的。任何创建的测试几乎肯定能发现故障。

有限状态机 (FSM)

什么是FSM？

有限状态机（FSM）是一个描述软件在执行过程中变量如何被修改的图。

概念含义示例
节点 (State)状态，表示关键变量的值的集合"运行中"状态
边 (Transition)转换，状态的可能变化从"运行"到"停止"

FSM的应用场景

  • 嵌入式和控制软件（电子设备）
  • 抽象数据类型
  • 编译器和操作系统
  • Web应用程序
  • UML状态图、自动机、状态表(SCR)、Petri网
FSM的局限 — 对于状态数量很大的程序（如GUI），FSM可能不实用。

FSM的标注

FSM可以标注不同类型的动作：

转换动作 (Actions on transitions)
在转换发生时执行的动作。
入口动作 (Entry actions)
进入状态时执行的动作。
出口动作 (Exit actions)
离开状态时执行的动作。
前置条件/守卫 (Preconditions/Guards)
转换必须满足的条件。
触发事件 (Triggering events)
导致转换发生的变量变化。

电梯示例

打开电梯门的转换：前置条件 elevSpeed = 0，触发事件 openButton = pressed。

FSM覆盖标准

覆盖标准含义
节点覆盖 (State Coverage)执行每个状态
边覆盖 (Transition Coverage)执行每个转换
边对覆盖 (Transition-Pair)执行每对转换
数据流覆盖对守卫或扩展状态计算def和use

FSM的构造方法

三种构造方法

不经验的开发者的第一反应是画CFG并连接它们。这实际上不是FSM。

存在的问题：
  • 方法必须返回到正确的调用点——隐式非确定性
  • 需要实现才能构建图
  • 图不能扩展

有经验的程序员可能将方法映射到状态。但这些实际上不是状态。

存在的问题：
  • 主观——不同的测试者得到不同的图
  • 需要深入了解实现
  • 需要详细设计

更机械的方法。首先识别所有状态变量，然后选择相关的。理论上，状态变量值的每种组合定义一个不同的状态。实践中，必须识别属于同一状态的值的范围。

推荐方法： 状态变量建模是最系统、最可重复的FSM构造方法，它不需要实现细节，且可以扩展到大型系统。

完整案例：Watch手表程序

程序代码

public class Watch {
  enum Mode { TIME, STOPWATCH, ALARM }
  Mode mode = Mode.TIME;
  Time currentTime;
  Time alarmTime;

  void doTransition(Event e) {
    if (mode == Mode.TIME) {
      if (e == B2) changeTime();
      if (e == B1) mode = Mode.STOPWATCH;
    } else if (mode == Mode.STOPWATCH) {
      if (e == B1) mode = Mode.ALARM;
    } else if (mode == Mode.ALARM) {
      if (e == B1) mode = Mode.TIME;
    }
  }

  void changeTime() {
    currentTime.incrementMinute();
  }
}
Java

状态变量分析

变量类型取值范围
mode状态变量TIME, STOPWATCH, ALARM
hourTime类1..12
minuteTime类0..59

如果直接组合，Time类有 12 × 60 = 720 个状态，加上mode的3个值共 2160 个状态，显然太多。

需要将值合并为范围：

  • hour: 1..11, 12
  • minute: 0, 1..59
  • Time: 12:00, 12:01..12:59, 01:00..11:59

层次化FSM

当一个类使用另一个类时（如Watch使用Time），可以构建层次化FSM。外层FSM（Watch的mode状态）包含内层FSM（Time的状态）。

FSM测试的优缺点

✅ 优点

  • 测试可以在实现之前设计
  • 分析FSM比分析源码容易得多

❌ 缺点

  • 一些实现决策未在FSM中建模
  • 由于FSM构造的主观性，结果可能有差异
  • 测试必须"映射"到程序的实际输入
