3.6 用例 活动图 场景测试

UML用例

UML用例常用于表达软件需求，帮助表达计算机应用的工作流程。用例图由参与者(Actor)和用例(Use Case)组成。

简单的用例图（如ATM用户和ATM系统）本身对测试不太有用，需要进行详细描述(Elaboration)。

用例详细描述

用例通常先以文本形式详细描述，包括：

  • 摘要 (Summary) — 用例的简要描述
  • 参与者 (Actor) — 使用软件的人员或组件
  • 前置条件 (Precondition) — 执行前必须满足的条件
  • 描述 (Description) — 详细的操作步骤
  • 替代方案 (Alternatives) — 执行过程中的选择和条件
  • 后置条件 (Postcondition) — 执行后的预期状态

活动图转换

从用例到活动图

Beizer提出"事务流图"(Transaction Flow Graph)，UML活动图(Activity Diagram)与之类似。

活动图表示活动之间的流程：

活动图元素对应内容图元素
动作状态 (Action State) 用例描述中的操作步骤 节点
顺序分支 (Sequential Branch) 替代方案中的选择和条件 节点
流 (Flow) 步骤之间的流转 边

活动图通常有以下特点：

  • 很少有环路
  • 简单的谓词
  • 没有明显的DU对

ATM取款案例

ATM取款用例

  1. 客户插入ATM卡到读卡器
  2. 系统识别卡号
  3. 系统提示输入PIN
  4. 客户输入PIN
  5. 系统检查卡的有效期和是否挂失
  1. 系统验证PIN是否匹配
  2. 系统查找可访问的账户
  3. 系统显示账户并提示选择交易类型
  4. 客户选择取款，选择账户，输入金额
  5. 系统验证账户、余额、每日限额、ATM余额
  1. 系统发放现金
  2. 系统打印收据
  3. 系统弹出卡
  4. 系统显示欢迎信息

替代方案

  • 卡无法识别：弹出卡，显示欢迎信息
  • 卡过期：没收卡，显示欢迎信息
  • 卡挂失/被盗：没收卡，显示欢迎信息
  • PIN不匹配：提示重新输入
  • PIN三次错误：没收卡，显示欢迎信息
  • 账户无效：显示错误，弹出卡
  • 超过每日限额：显示道歉，弹出卡
  • ATM余额不足：显示道歉，弹出卡
  • 客户按取消：取消交易，弹出卡

覆盖活动图

覆盖标准

节点覆盖 (Node Coverage)
执行每个活动节点至少一次。从节点标签和谓词推导输入值。
边覆盖 (Edge Coverage)
执行每个分支至少一次。
场景测试 (Scenario Testing)
场景是通过用例活动图的完整路径。应该对用户有语义意义。如果路径有限，使用指定路径覆盖(SPC)。
活动图通常结构简单（很少有环路），因此指定路径覆盖是合理的选择。场景应定义得使SPC至少包含边覆盖。

用例测试总结

  • 用例定义在需求级别
  • UML活动图将用例编码为图
  • 图通常结构相当简单
  • 基于需求的测试可以使用图覆盖
  • 手动操作相对简单直接
  • 指定路径覆盖对这些图很有意义
