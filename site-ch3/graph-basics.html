图基础与覆盖标准

Graph Coverage Basics — 图的形式化定义、结构化覆盖标准（NC、EC、EPC、PPC）以及数据流覆盖标准（ADC、AUC、ADUPC）。

3.1 3.2 NC EC PPC ADC

图的定义与基本概念

什么是图覆盖？

图是软件测试中最常用的结构。图可以来自多种来源：控制流图、设计结构、有限状态机（FSM）和状态图、用例等。测试通常旨在以某种方式"覆盖"图——即确保测试用例的执行路径遍历图中的节点、边或更长的子路径。

图的形式化定义

一个图 G 定义为四元组 <N, N₀, Nf, E>，其组成部分如下：

组成说明示例
节点集合 N 图中的所有节点 {1, 2, 3, 4}
初始节点集合 N₀ 测试路径的起始节点 {1}
终结节点集合 Nf 测试路径的结束节点 {4}
边集合 E 节点间的连接关系 {(1,2), (1,3), (2,4), (3,4)}
有效图的条件： N、N₀ 和 Nf 必须非空，E 可以为空（孤立节点的情况）。N₀ 和 Nf 中的节点必须属于 N。

路径相关概念

概念定义示例
路径 (Path) 节点和边的交替序列，如 [n₁, e₁, n₂, e₂, …, nₖ] [1, 2, 4]
测试路径 (Test Path) 从初始节点到终结节点的路径 [1, 2, 4]
访问 (Visit) 测试路径 p 访问节点 n（或边 e），当 n（或 e）出现在 p 中 节点或边
游历 (Tour) 测试路径 p 游历子路径 q，当 q 是 p 的子路径 子路径
可达 (Reach) 从节点 nᵢ 到节点 n 存在路径，记作 reach(n) reach(n)

语法可达 vs 语义可达

🔍 语法可达 (Syntactic Reach)

从图的结构上存在对应的子路径。可以通过图的结构分析来判断，不需要运行程序。

  • 基于图的拓扑结构
  • 静态分析即可判断
  • 可能存在语法可达但语义不可达的情况

🧠 语义可达 (Semantic Reach)

存在测试用例能够执行对应的子路径。需要考虑程序的实际行为和条件约束。

  • 基于程序的实际执行语义
  • 需要构造满足条件的测试输入
  • 不可行路径是语义不可达的

SESE 图

SESE (Single-Entry, Single-Exit) 图：所有测试路径从单个节点开始，在另一个节点结束。这种图结构在实际程序中非常常见，例如一个函数的控制流图通常就是 SESE 图。典型的 SESE 图——双菱形图 (Double-diamond graph)——有 4 条测试路径。

结构化覆盖标准

结构化覆盖标准定义了测试需要覆盖图中哪些元素。以下六种标准从弱到强排列：

Node Coverage (NC) — 节点覆盖

定义： 测试需求集 TR 包含图 G 中每个可达节点。

NC 等价于访问图中每个可达节点至少一次。这是最基本的图覆盖标准，确保程序中的每条语句（对应节点）至少被执行一次。

测试数量上界： 节点数。例如图 G = <{1,2,3}, {1}, {3}, {(1,2),(2,3)}>，TR = {1, 2, 3}，测试路径 = [1, 2, 3]。

Edge Coverage (EC) — 边覆盖

定义： 测试需求集 TR 包含图 G 中每条可达的长度不超过 1 的路径。

EC 要求覆盖每条可达边。定义中"长度不超过 1"的表述允许处理只有一个节点、没有边的特殊情况。

EC 比 NC 稍强： 在一对节点间存在一条边和另一条更长子路径时，NC 和 EC 的要求才会不同（例如 if-else 语句的两个分支）。

EC ⊇ NC： 边覆盖包含节点覆盖，因为覆盖了所有边必然覆盖了这些边涉及的所有节点。

Edge-Pair Coverage (EPC) — 边对覆盖

定义： 测试需求集 TR 包含图 G 中每条可达的长度不超过 2 的路径。

EPC 要求覆盖所有边对（长度为 2 的子路径）。定义中"长度不超过 2"用于包含少于 2 条边的图。

EPC ⊇ EC： 边对覆盖包含边覆盖，因为覆盖了所有长度为 2 的路径必然覆盖了所有长度为 1 的路径（边）。

Complete Path Coverage (CPC) — 完全路径覆盖

定义： 测试需求集 TR 包含图 G 中所有路径。

CPC 要求覆盖图中的所有可能路径。这是最强的结构化覆盖标准。

CPC 在有环路的图中不可行： 如果图中存在环路，则路径数量是无限的，无法实现完全路径覆盖。实践中只能使用较弱但可行的标准。

Specified Path Coverage (SPC) — 指定路径覆盖

定义： 测试需求集 TR 包含一个给定的路径集合 S，其中 S 作为参数提供。

SPC 由测试者决定覆盖哪些路径，因此结果具有主观性。测试人员根据经验和需求选择一组重要的路径进行覆盖。

Prime Path Coverage (PPC) — 主路径覆盖

定义： 测试需求集 TR 包含图 G 中每条主路径。

PPC 是最优雅的结构化覆盖标准，要求覆盖所有主路径。主路径是不可再延伸的简单路径，详见下一节。

PPC 包含关系： PPC ⊇ NC 且 PPC ⊇ EC。PPC 几乎但不完全包含 EPC（自环边的特殊情况）。

覆盖标准对比

标准覆盖对象包含关系
NC 每个可达节点 —
EC 每条可达边（长度 ≤ 1） ⊇ NC
EPC 每对可达边（长度 ≤ 2） ⊇ EC
CPC 所有路径 ⊇ EPC
PPC 所有主路径 ⊇ EC, almost ⊇ EPC

简单路径与主路径

简单路径 (Simple Path)

从节点 nᵢ 到节点 nⱼ 的路径是简单路径，当且仅当没有节点重复出现（除了首尾节点可能相同的情况）。简单路径没有内部环路。注意：环本身也是一条简单路径。

主路径 (Prime Path)

主路径是不作为其他简单路径的真子路径出现的简单路径，即不能再延伸的简单路径。主路径是长度最长的简单路径。

主路径查找方法

按路径长度从短到长枚举所有简单路径，标记终止路径（!）和循环路径（*），然后找出不被其他简单路径包含的路径作为主路径。

以下以一个包含节点 {1, 2, 3, 4, 5, 6, 7} 的图为例，展示查找过程：

// 长度为 0 的简单路径（单个节点）
[1], [2], [3], [4], [5], [6], [7]

// 长度为 1 的简单路径（单条边）
[1, 2], [1, 3], [2, 3], [3, 4], [3, 5], [4, 7], [5, 7], [5, 6], [6, 5]

// 长度为 2 的简单路径
[1, 2, 3], [1, 3, 4], [1, 3, 5], [2, 3, 4], [2, 3, 5],
[3, 4, 7], [3, 5, 7], [3, 5, 6], [5, 6, 5]*, [6, 5, 6]*, [6, 5, 7]

// 长度为 3 的简单路径
[1, 2, 3, 4], [1, 2, 3, 5], [1, 3, 4, 7], [1, 3, 5, 7], [1, 3, 5, 6],
[2, 3, 4, 7], [2, 3, 5, 7], [2, 3, 5, 6], [3, 5, 6, 5]*, [6, 5, 6, 5]*

// 长度为 4 的简单路径
[1, 2, 3, 4, 7]!, [1, 2, 3, 5, 7]!, [1, 2, 3, 5, 6],
[1, 3, 5, 6, 5]*, [2, 3, 5, 6, 5]*

// 长度为 5 的简单路径
[1, 2, 3, 5, 6, 5]*, [1, 3, 5, 6, 5]*

// 共 38 条简单路径
Simple Paths

从中识别出 9 条主路径：

// 主路径（不能再延伸的简单路径）
[1, 2, 3, 4, 7]    // 终结于终结节点，无法延伸
[1, 2, 3, 5, 7]    // 终结于终结节点，无法延伸
[1, 2, 3, 5, 6]    // 6→5 构成环路，但 [1,2,3,5,6,5] 也是简单路径 → 需检查
[1, 3, 4, 7]       // 终结于终结节点，无法延伸
[1, 3, 5, 7]       // 终结于终结节点，无法延伸
[1, 3, 5, 6]       // 6→5 构成环路
[6, 5, 7]          // 终结于终结节点，无法延伸
[6, 5, 6]          // 环路，无法延伸（再延伸会重复内部节点）
[5, 6, 5]          // 环路，无法延伸
Prime Paths
统计： 38 条简单路径中仅 9 条是主路径。主路径覆盖显著减少了需要测试的路径数量，同时保持了较高的覆盖强度。

PPC 与 EPC 的关系

如果一个节点 n 有指向自己的边（自环），EPC 需要覆盖路径 [n, n, m] 和 [m, n, n]，但这些路径中节点 n 重复出现，不是简单路径（也不是主路径）。因此 PPC 不完全包含 EPC。这是 PPC 与 EPC 之间唯一不满足包含关系的情况。

游历、侧访与绕道

三种游历方式

直接游历 (Direct Tour)
测试路径 p 直接游历子路径 q，当 q 是 p 的连续子路径。这是最严格的游历方式。
侧访游历 (Tour with Sidetrips)
测试路径 p 侧访游历子路径 q，当且仅当 q 中的每条边也按相同顺序出现在 p 中。游历可以包含侧访——从子路径的某个节点出发走一段岔路后，回到同一节点继续。侧访要求回到岔道的起点。
绕道游历 (Tour with Detours)
测试路径 p 绕道游历子路径 q，当且仅当 q 中的每个节点也按相同顺序出现在 p 中。游历可以从节点 nᵢ 开始绕道，只要回到 nᵢ 的后继节点继续即可。绕道不要求回到岔道的起点，但要求节点顺序一致。

不可行测试需求

不可行的测试需求无法被满足，常见的不可行情况包括：

  • 不可达语句（死代码）： 没有任何输入能执行到的代码
  • 矛盾条件的子路径： 例如同时要求 X > 0 且 X < 0 的路径
实践建议 — 最佳努力游历 (Best Effort Touring)： 先尽可能用直接游历满足测试需求，再允许使用侧访来满足剩余的（可能因不可行性而难以直接满足的）需求。这种策略在实际项目中非常实用。

环路路径

环路路径 (Round-Trip Path)
起始和结束于同一节点的主路径，即首尾节点相同的主路径。
简单环路覆盖 (SRTC — Simple Round Trip Coverage)
TR 包含每个可开始和结束环路路径的可达节点的至少一条环路路径。
完全环路覆盖 (CRTC — Complete Round Trip Coverage)
TR 包含每个可达节点的所有环路路径。
注意： SRTC 和 CRTC 不包含边对覆盖（EPC）、边覆盖（EC）或节点覆盖（NC）。它们只关注环路路径，不保证覆盖所有节点或边。因此在使用时通常需要与其他覆盖标准配合。

数据流覆盖标准

数据流覆盖标准关注变量的定义和使用之间的关系，比单纯的结构覆盖更能发现与变量相关的缺陷。

基本概念

概念定义示例
定义 (def) 变量值被存入内存的位置 赋值语句左侧、方法形参、输入语句
使用 (use) 变量值被访问的位置 赋值语句右侧、条件判断、方法实参、输出语句
def(n) 或 def(e) 节点 n 或边 e 定义的变量集合 def(1) = {x, sum}
use(n) 或 use(e) 节点 n 或边 e 使用的变量集合 use(5) = {x, length}

DU 对与 DU 路径

DU 对 (DU-pair)
变量 v 在位置 lᵢ 定义、在位置 lⱼ 使用的位置对 (lᵢ, lⱼ)。
def-clear 路径
从 lᵢ 到 lⱼ 的路径关于变量 v 是 def-clear 的，当且仅当 v 在路径的任何节点或边上没有被重新赋值（即没有新的定义）。
到达 (Reach)
如果从 lᵢ 到 lⱼ 存在关于 v 的 def-clear 路径，则 lᵢ 处对 v 的定义到达 lⱼ 处的使用。
du 路径 (du-path)
关于 v 从定义到使用的简单 def-clear 子路径。du 路径是简单路径，没有内部环路。
du(nᵢ, nⱼ, v)
从 nᵢ 到 nⱼ 关于 v 的 du 路径集合。
du(nᵢ, v)
从 nᵢ 开始的关于 v 的所有 du 路径集合。

三种数据流覆盖标准

All-Defs Coverage (ADC) — 全定义覆盖

定义： 对于每组 du 路径集合 S = du(n, v)，TR 包含 S 中至少一条路径 d。

ADC 确保每个定义到达至少一个使用。这是最弱的数据流覆盖标准，只需保证每个变量定义都被使用过（不会定义了变量却不使用）。

All-Uses Coverage (AUC) — 全使用覆盖

定义： 对于每组到使用的 du 路径集合 S = du(nᵢ, nⱼ, v)，TR 包含 S 中至少一条路径 d。

AUC 确保每个定义到达所有可能的使用。比 ADC 更强，因为它要求每个定义-使用对都被至少覆盖一次。

AUC ⊇ ADC： 全使用覆盖包含全定义覆盖，因为覆盖了所有使用必然覆盖了所有定义。

All-DU-Paths Coverage (ADUPC) — 全DU路径覆盖

定义： 对于每组 S = du(nᵢ, nⱼ, v)，TR 包含 S 中的每条路径 d。

ADUPC 覆盖所有定义到使用之间的所有路径。这是最强的数据流覆盖标准，要求同一个 DU 对之间的所有不同 du 路径都被覆盖。

ADUPC ⊇ AUC： 全DU路径覆盖包含全使用覆盖。

数据流覆盖标准对比

标准覆盖对象包含关系
ADC 每个定义至少到达一个使用 —
AUC 每个定义到达所有使用 ⊇ ADC
ADUPC 覆盖所有 DU 路径 ⊇ AUC

结构化覆盖标准关注图的节点和边，数据流覆盖标准关注变量的定义和使用。两者结合可以更全面地测试软件：结构覆盖确保代码的每个部分都被执行，数据流覆盖确保变量的值被正确地传递和使用。
