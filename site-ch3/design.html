§ 3.4 调用图 耦合DU 集成测试

调用图

调用图基础

调用图(Call Graph)是结构化设计测试中最常见的图。节点是单元（Java中的方法），边是对单元的调用。

覆盖标准 含义 说明
节点覆盖 调用每个方法至少一次 方法覆盖
边覆盖 执行每个调用至少一次 调用覆盖

类调用图的局限

类中各个方法可能互不调用（如Stack类的push、pop、isEmpty），因此节点覆盖和边覆盖效果不佳，需要其他测试方法。

面向对象调用覆盖

继承与多态

在继承层次结构中，类本身不可执行，需要创建对象。因此调用图需要基于实例化对象来构建。

OO调用覆盖标准

OO调用覆盖 (OO Call Coverage)
TR包含为类层次结构中每个类实例化的对象的调用图中的每个可达节点。
OO对象调用覆盖 (OO Object Call Coverage)
TR包含为类层次结构中每个类的每个实例化对象的调用图中的每个可达节点。

耦合数据流

基本概念

设计层面的数据流耦合比控制流耦合更复杂。当值被传递时，它们"改变名称"。有许多不同的数据共享方式。

术语 含义 示例
调用者 (Caller) 调用其他单元的单元 方法A调用方法B
被调用者 (Callee) 被调用的单元 方法B
调用点 (Callsite) 调用出现的语句或节点 B(x)
实参 (Actual Parameter) 调用者中的变量 x
形参 (Formal Parameter) 被调用者中的变量 y

过程间DU对

如果关注接口，只需考虑调用前变量的最后定义(last-def)和调用后/单元内的首次使用(first-use)。

Last-def
定义变量x的节点集合，从该节点通过调用点到另一单元中的使用存在def-clear路径。可以是从调用者到被调用者（参数或共享变量），也可以是从被调用者到调用者（返回值）。
First-use
使用变量y的节点集合，从调用点到这些节点存在def-clear且use-clear的路径。

示例

以F和G两个方法为例：

// Caller: F
X = 14;
...
y = G(x);
...
print(y);

// Callee: G(a)
print(a);
...
b = 42;
return(b);

Last Defs: 节点2, 3（在F中定义x）

First Uses: 节点11, 12（在G中使用a）

DU对: (A, x, 2)→(B, y, 11), (A, x, 2)→(B, y, 12), (A, x, 3)→(B, y, 11), (A, x, 3)→(B, y, 12)

三种耦合覆盖标准

All-Coupling-Def覆盖
每个last-def到至少一个first-use。
All-Coupling-Use覆盖
每个last-def到每个first-use。
All-Coupling-du-Paths覆盖
每个last-def到每个first-use之间的每条简单路径。

完整案例：Quadratic程序

程序代码

public class Quadratic {
  public static void main(String[] args) {
    double X, Y, Z;
    double Root1, Root2;
    boolean ok;
    // read X, Y, Z from input
    if (X == 0) {
      ok = false;
    } else {
      ok = Root(X, Y, Z, Root1, Root2);
    }
    if (ok) {
      print(Root1, Root2);
    } else {
      print("No real roots");
    }
  }

  public static boolean Root(double A, double B, double C,
                             double Root1, double Root2) {
    boolean Result;
    double D;
    D = B * B - 4 * A * C;
    if (D < 0) {
      Result = false;
    } else {
      Root1 = (-B + Math.sqrt(D)) / (2 * A);
      Root2 = (-B - Math.sqrt(D)) / (2 * A);
      Result = true;
    }
    return Result;
  }
}

Coupling DU-pairs分析

方法对 变量 语句 说明
(main, Root) X → A 15→38 参数传递
(main, Root) Y → B 16→38 参数传递
(main, Root) Z → C 17→38 参数传递
(main, Root) X → A 21→38 参数传递(else分支)
(main, Root) Y → B 22→38 参数传递(else分支)
(main, Root) Z → C 23→38 参数传递(else分支)
(Root, main) Root1 44→28 返回值
(Root, main) Root2 45→28 返回值
(Root, main) Result → ok 41→26 返回值
(Root, main) Result → ok 46→26 返回值

耦合数据流注意事项

  • 只考虑在被调用者中使用或定义的变量。
  • 类和全局变量的隐式初始化。
  • 传递性DU对（A调用B，B调用C，变量在A中定义在C中使用）代价太高。
  • 数组：对一个元素的引用被视为对所有元素的引用。

现代演进

传统静态调用图无法处理多态、反射、动态分派。

现代方法：Dynamic Call Graph（运行时记录实际调用关系）、Hybrid Approach（结合静态分析和运行时信息）。

工程实践：Distributed Tracing（如 Jaeger、Zipkin）在微服务架构中构建跨服务调用链。

演进路线：Last-def/First-use → API-level Data Flow → Taint Analysis（污点分析）。

核心区别：DU基于枚举路径，Taint基于跟踪传播——不需要枚举所有路径，而是追踪数据从源（source）到汇（sink）的传播过程。

传统概念在微服务架构中的映射：

  • Caller/Callee → Service A / Service B
  • Actual Parameter → JSON payload（请求体）
  • Return value → HTTP response（响应体）

Service Dependency Graph (SDG) 用于微服务架构中的依赖分析和测试覆盖。

传统 Node/Edge/DU Coverage 关注路径执行，现代更关注测试的有效性。

实践趋势：

  • Fuzzing — Data-flow-guided fuzzing，利用数据流信息引导模糊测试生成更有效的输入。
  • Mutation Testing — 通过变异测试评估测试套件的缺陷检测能力。
  • Interface/API Testing — 聚焦接口契约和数据交换的正确性。
