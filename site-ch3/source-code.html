源代码图覆盖

Graph Coverage for Source Code — 将图覆盖标准应用于程序源码，通过控制流图(CFG)覆盖各种控制结构，并进行源码级数据流分析。

3.3 CFG 控制流图 数据流

控制流图 (CFG)

什么是控制流图？

控制流图（Control Flow Graph, CFG）通过描述控制结构来建模方法的所有执行。CFG是图覆盖标准在源代码中最常见的应用。

CFG 的基本组成

组成说明示例
节点 (Node) 语句或语句序列（基本块） 一个赋值语句
边 (Edge) 控制转移 分支判断后的跳转
基本块 (Basic Block) 一串语句，如果第一个被执行，则所有语句都会被执行（无分支） 循环体

CFG与覆盖标准的对应关系

图覆盖标准源代码对应
节点覆盖 (NC) 执行每条语句
边覆盖 (EC) 执行每个分支
环路覆盖 覆盖循环结构
数据流覆盖 增强CFG，标记def和use

典型控制结构的CFG

if 语句

if语句产生两条边——条件为真和条件为假的分支。

// 源代码
if (x > 0) {
    y = x;
} else {
    y = -x;
}

// CFG 表示
       [1: if (x > 0)]
          /        \
     true          false
        /            \
[2: y = x]      [3: y = -x]
        \            /
         [4: 之后]
CFG

if-return 语句

当if中包含return时，产生一个直接到结束节点的边。

// 源代码
if (x > 0) {
    return x;
}
y = -x;

// CFG 表示
       [1: if (x > 0)]
          /        \
     true          false
        /            \
[return x]      [2: y = -x]
                        |
                   [EXIT]
CFG

while 和 for 循环

while和for循环产生一个回到循环开始的边，以及一个退出循环的边。for循环中的增量操作隐含在循环结构中。

// while 循环 CFG
       [1: while (i < n)]
          /        \
     true          false
        /            \
   [2: body]       [3: 之后]
        |
        +--→ [1]

// for 循环 CFG
       [1: i = 0]
            |
       [2: if (i < n)]
          /        \
     true          false
        /            \
   [3: body]       [5: 之后]
        |
   [4: i++]
        |
        +--→ [2]
CFG

do-while 循环

do-while循环至少执行一次循环体，然后判断条件。

// do-while CFG
       [1: body]
            |
       [2: if (cond)]
          /        \
     true          false
        /            \
      [1]         [3: 之后]
CFG

break 和 continue

break产生直接退出循环的边，continue产生回到循环判断的边。

// break / continue CFG
       [1: while (cond)]
          /    |     \
     true   brk    false
        /      |       \
   [2: body]  |      [4: 之后]
    /    \     |
[continue] [break]
   |          |
  [1]       [4]
CFG

switch/case 结构

每个case产生一条边，没有break的case会穿透到下一个case。

// switch/case CFG
       [1: switch (x)]
       /      |      \
   case 1  case 2  default
      |       |       |
   [2: ...] [3: ...] [4: ...]
      \       |       /
       [5: 之后]
CFG

try-catch 异常处理

try块中的每个可能抛出异常的语句都产生到对应catch块的边。

// try-catch CFG
       [1: try { stmt1 }]
          /        \
     normal      exception
        /            \
[2: stmt2]      [4: catch { ... }]
        \            /
         [3: 之后]
CFG

源码级数据流覆盖

源码中的 def 和 use

在源代码层面，def和use的识别规则如下：

类型规则示例
def x出现在赋值左侧 x = 44;
def x是方法的形参 void foo(int x)
def x是程序的输入 read(x)
use x出现在赋值右侧 y = x + 1;
use x出现在条件判断中 if (x > 0)
use x是方法的实参 foo(x)
use x是程序的输出 print(x)
注意：如果def和use出现在同一节点，只有当def发生在use之后且节点在循环中时，才构成DU对。

完整案例：Stats程序

程序代码

public static void computeStats(int[] numbers) {
    int length = numbers.length;
    double med, var, sd, mean, sum, varsum;
    sum = 0;
    for (int i = 0; i < length; i++) {
        sum += numbers[i];
    }
    med = numbers[length / 2];
    mean = sum / (double) length;
    varsum = 0;
    for (int i = 0; i < length; i++) {
        varsum = varsum + ((numbers[i] - mean) * (numbers[i] - mean));
    }
    var = varsum / (length - 1.0);
    sd = Math.sqrt(var);
    System.out.println("length: " + length);
    System.out.println("mean: " + mean);
    System.out.println("median: " + med);
    System.out.println("variance: " + var);
    System.out.println("standard deviation: " + sd);
}
Java

节点语句

2int i = 0;（循环初始化）
3if (i < length)（循环判断）
4sum += numbers[i]; i++;（循环体）

Edge Coverage (EC)

测试需求：覆盖CFG中的每条边。

边集合： (1,2), (2,3), (3,4), (3,5), (4,3), (5,6), (6,7), (6,8), (7,6)

测试路径：

EC Test Path

对于含循环的程序，一个测试用例即可覆盖所有边（循环至少执行一次即可覆盖循环边和退出边）。

Edge-Pair Coverage (EPC)

测试需求：覆盖CFG中所有长度为2的边序列（边对）。

边对集合：

  • (1,2,3), (2,3,4), (2,3,5)
  • (3,4,3), (4,3,4), (4,3,5)
  • (3,5,6), (5,6,7), (5,6,8)
  • (6,7,6), (7,6,7), (7,6,8)

测试路径：

// 两个测试用例覆盖所有边对 ✓
EPC Test Paths

Prime Path Coverage (PPC)

测试需求：覆盖CFG中所有主路径（不能再延伸的简单路径）。

主路径集合：

  • [2, 3, 5, 6, 8] — 两个循环都执行0次（不可行，因为循环至少判断一次后进入）
  • [3, 4, 3] — 第一个循环内部
  • [4, 3, 5, 6, 8] — 第一个循环退出到结束
  • [6, 7, 6] — 第二个循环内部
  • [7, 6, 8] — 第二个循环退出到结束
  • [4, 3, 4] — 第一个循环重复
  • [7, 6, 7] — 第二个循环重复

测试路径：

// 测试用例 1: numbers = [10] (length=1)

// 测试用例 2: numbers = [10, 20, 30] (length=3)
Path 2: [1, 2, 3, 4, 3, 4, 3, 4, 3, 5, 6, 7, 6, 7, 6, 7, 6, 8]

// 两个测试用例覆盖所有主路径 ✓
PPC Test Paths
PPC ⊇ EPC ⊇ EC： 主路径覆盖包含边对覆盖，边对覆盖包含边覆盖。对于Stats程序，最少需要2个测试用例即可满足PPC。

现代趋势

传统CFG+def-use分析 → SSA（静态单赋值形式）直接编码def-use关系 → Dataflow IR直接消除控制依赖。

本质：把数据流从"分析结果"变成"程序表示本身"。LLVM IR、MLIR等现代编译器中间表示已经将数据流关系显式编码在程序表示中。

Datalog/SMT方法，如Rust编译器中的Polonius借用检查器，将def-use转换为逻辑规则，通过约束求解来验证数据流性质。

这种方法可以更精确地处理复杂的生命周期和借用关系，减少误报。

从"变量是否被定义使用"到"信息是否非法传播"。Taint分析追踪不可信数据（污点源）是否流向安全敏感位置（污点汇），是数据流分析在安全领域的重要应用。

典型工具：FlowDroid（Android污点分析）、Joana（Java信息流分析）。

结合Symbolic Execution的path-sensitive分析，不仅追踪数据流的定义和使用，还考虑路径条件对数据流可达性的影响。

可以消除不可行路径上的误报，提高分析精度，但代价是更高的计算复杂度。

Trace-based dataflow inference和runtime taint tracking，通过运行时监控和机器学习技术动态推断数据流关系。

适用于动态语言和高度反射的系统，传统静态分析难以处理的场景。
