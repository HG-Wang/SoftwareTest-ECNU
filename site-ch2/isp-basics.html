输入空间划分基础

ISP Basics — 输入域、划分的性质、输入域建模的五步方法、基于接口和基于功能的建模方法。

ISP 输入域建模 特征 块

输入域

什么是输入域？

输入域包含程序所有可能的输入。即使是一个小程序，其输入域也大到几乎无限。测试人员不可能穷举所有输入，因此需要系统的方法来选择有限的测试用例。

输入参数的来源

输入参数定义了输入域的范围，来源包括：

来源说明示例
方法参数 函数或方法的形参 triang(int a, int b, int c)
文件数据 从文件读取的输入 配置文件、数据文件
全局变量 程序中可访问的全局状态 环境变量、全局标志
用户输入 用户通过界面输入的数据 表单输入、命令行参数
核心思想： 域被划分为区域，从每个区域中选取至少一个值进行测试。这就是"输入空间划分"的含义。

划分的性质

形式化定义

一个划分 q 定义块集合 Bq = {b1, b2, ..., bQ}，必须满足两个性质：

🔒 互斥性 (Pairwise Disjoint)

bi ∩ bj = ∅, ∀i≠j

任意两个不同的块之间没有交集。每个输入值只能属于一个块。

✅ 完备性 (Complete)

∪b = D

所有块的并集等于整个输入域D。每个输入值都必须属于某个块。

选择值的假设

ISP假设：从每个块中选择的值是同等有用的——即块内的值在发现缺陷方面是等价的。这个假设并不总是成立，但它是ISP方法的基础。

示例：文件排序的问题

考虑一个文件排序程序，我们定义以下特征：

  • 特征1：文件长度 — 块：{长度=0}, {长度=1}, {长度>1}
  • 特征2：文件顺序 — 块：{已排序}, {逆序}, {随机}
  • 特征3：元素唯一性 — 块：{无重复}, {有重复}

问题：长度为1的文件同时属于"已排序"、"逆序"和"无重复"三个块——这违反了互斥性。

解决方案： 每个特征只关注一个属性。例如"长度"特征只关注文件有多少行，"顺序"特征只关注元素的排列方式。避免让一个特征隐含多个属性。

输入域建模五步法

输入域建模（Input Domain Modeling, IDM）是ISP的核心，包含以下五个步骤：

  1. 识别可测试功能

    确定要测试的程序功能或方法。例如对于三角形分类程序，可测试功能是 triang(int a, int b, int c) 方法。

  2. 找出所有参数

    列出功能的所有输入参数及其类型。例如 triang 有三个 int 类型参数 a、b、c。

  3. 建模输入域 最具创造性

    为每个参数定义特征，再将每个特征划分为块。这是整个过程中最具创造性的步骤，需要结合领域知识和测试经验。

  4. 应用测试准则选择组合

    使用组合覆盖标准（ACoC、ECC、PWC、BCC等）从各块中选择组合。

  5. 精化组合为测试输入

    将选中的块组合转化为具体的测试输入值。

基于接口的方法

基本思路

基于接口的方法机械地考虑每个参数，为每个参数定义特征和块。不考虑参数之间的关系。

示例：triang() 的基于接口建模

三角形分类方法 triang(int a, int b, int c)：

第一轮建模

每个参数与0的关系：

参数特征块
a与0的关系{a>0}, {a=0}, {a<0}
b与0的关系{b>0}, {b=0}, {b<0}
c与0的关系{c>0}, {c=0}, {c<0}

全组合：3 × 3 × 3 = 27 种测试

精化建模

加入=1的特殊情况：

参数特征块
a与0和1的关系{a>1}, {a=1}, {a=0}, {a<0}
b与0和1的关系{b>1}, {b=1}, {b=0}, {b<0}
c与0和1的关系{c>1}, {c=1}, {c=0}, {c<0}

全组合：4 × 4 × 4 = 64 种测试

✅ 优点

  • 简单、机械，可部分自动化
  • 不需要深入理解程序行为
  • 适合参数独立的情况

❌ 缺点

  • 忽略参数间关系
  • 可能不完整（遗漏重要特征）
  • 可能产生不可行的测试组合

基于功能的方法

基本思路

基于功能的方法从程序的行为视图出发，考虑程序应该如何响应不同类型的输入。需要理解程序的功能和领域知识。

示例：triang() 的基于功能建模

从几何分类的角度，三角形可以分为：

特征块说明
三角形类型{scalene}三边不等
{isosceles}两边相等
{equilateral}三边相等
{invalid}不能构成三角形
注意互斥性： equilateral（等边三角形）同时也是 isosceles（等腰三角形）。需要精化使块互斥——例如将 isosceles 定义为"恰好两边相等"。

✅ 优点

  • 可融入领域和语义知识
  • 能发现更有意义的测试分区
  • 测试用例更贴近实际使用场景

❌ 缺点

  • 需要更多设计努力
  • 依赖测试人员的领域知识
  • 难以自动化
