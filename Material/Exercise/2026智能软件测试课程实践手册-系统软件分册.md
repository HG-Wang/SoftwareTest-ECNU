# 一、 概述

# 1.1 实践目标

本实践以风险驱动测试决策为核心方法。

测试的目标不是追求覆盖率最大化，也不是增加测试数量，而是在资源受限条件下回答：

⚫ 哪些风险真正值得优先控制？

⚫ 哪些问题可能产生严重后果？

⚫ 当测试数量有限时，应如何进行测试决策？

本实践选取来自工业级开源项目、具有真实控制流复杂度与输入边界风险特征的中等规模功能模块（约 1000 LOC）作为风险驱动测试决策训练对象，通过系统建模、风险分析与测试决策过程，训练在工程约束下进行理性测试取舍的能力。

# 1.2 实践主线

本实践采用统一决策路径：

系统建模 风险分析 测试决策 覆盖分析 决策反思

各阶段功能如下：

⚫ 系统建模：刻画程序功能语义、输入域与关键判定逻辑

风险分析：识别并排序可能影响系统质量的风险

⚫ 测试决策：在资源受限条件下进行资源分配

⚫ 覆盖分析：验证高风险结构是否被触达

⚫ 决策反思：反思风险排序与测试取舍的合理性

说明：覆盖率不是目标，而是用于检验测试决策是否触达关键风险结构的辅助证据。

# 1.3 实践范围说明

在软件系统中，风险通常由功能语义与程序结构共同形成：

⚫ 功能语义：功能行为与预期语义不一致，决定风险后果严重程度

⚫ 输入域：参数组合或边界条件决定风险触发入口

⚫ 关键判定逻辑：条件分支与状态路径决定风险传播路径。

# 语义决定风险价值，结构决定风险分布与触达方式。

系统建模阶段需要刻画输入域与关键判定逻辑，是因为测试决策必须基于结构特征判断风险分布，而非仅基于语义描述。

本实践聚焦于：

与测试决策直接相关的风险分析

⚫ 基于输入域与关键判定逻辑特征进行测试资源分配

本实践的风险分析依据附录 B 所给出的风险识别维度。

测试资源分配，是指在测试数量与测试设计精力受限条件下，优先覆盖高风险语义所对应的关键输入域与关键判定逻辑。

附录 A 用于说明不同风险类型通常对应的控制机制，帮助理解方法层面的风险控制逻辑，但不参与风险排序。

附录 E 展示静态分析、内存检测与变异测试等工具在工程中的补充作用，用于扩展风险控制视角，不改变主线。

# 1.4 能力目标

本实践模块旨在培养学生形成以下四类核心能力：

1. 系统建模能力:能识别被测对象的功能结构、输入空间与潜在失效模式。

2. 风险分析与排序能力:能基于影响与可能性进行风险识别与优先级判断。

3. 测试决策能力:能依据风险结构分配测试资源并制定测试侧重点。

4. 边界意识与方法迁移能力:能理解覆盖指标与风险控制的关系，识别方法在不同工程情境下的适用范围。

# 第一部分：系统建模

目的：为风险分析提供结构基础。

注意：建模用于定位可能引发风险的结构位置，而不是穷尽所有行为细节。

# 5.1.1 功能语义识别

简述被测程序的核心行为，包括：

⚫ 默认输出行为

⚫ 选项对输出行为的影响

⚫ 参数与输出关系

⚫ 换行与转义处理规则

要求：

⚫ 不超过 10 行文字

描述行为含义，而非实现细节

# 5.1.2 输入域识别

描述被测程序的输入构成：

⚫ 选项集合（如 -n、-e 等）

. 普通参数（字符串）

⚫ 选项与参数的组合关系

可采用：

⚫ 结构图

⚫ 或分层列表

# 5.1.3 关键判定逻辑识别

结合代码或行为说明，识别 3–5 个关键判定逻辑。

例如：

⚫ 是否存在 -n 选项

⚫ 是否启用转义处理

⚫ 参数是否为空

⚫ 多选项顺序是否影响行为

要求：每项用 1–2 行说明其对行为的影响

# 5.1.4 建模总结

# 回答：

⚫ 哪些输入维度可能引发语义偏差？

⚫ 哪些判定逻辑可能导致路径差异？

⚫ 哪些结构位置值得在风险分析阶段重点关注？

# 要求：

不得仅写“可能出错”类描述

⚫ 风险描述

⚫ 严重性评分

⚫ 发生概率评分

注意：排序解释质量比排序结果本身更重要。

# 第三部分：测试决策

目的：在资源受限条件下进行测试资源分配。

每个测试必须包含：

⚫ 预期输出

⚫ 对应风险编号

⚫ 设计依据

设计依据应包含两个层面：

# （1）构造技术依据

⚫ 基于输入域划分选择代表性分区

⚫ 或基于关键边界值构造

说明该测试如何触达目标结构。

# （2）资源分配依据

⚫ 对应风险的优先级

⚫ 结构复杂度或触发难度

说明为何在测试数量受限条件下优先选择该测试。

说明：

⚫ 输入域划分与边界值分析属于测试构造技术；

# 附录 B 风险识别维度

本部分提供风险识别方向，用于辅助分析。

说明：

仅为思考方向，并非风险条目

⚫ 不要求每个维度必须识别出风险

⚫ 不得直接抄写维度作为风险条目

⚫ 每条风险必须结合代码与使用场景具体描述

# B.1 功能行为风险（Functional Behavior Risk）

指程序在正常使用场景下可能出现的功能性偏差。

思考方向：

⚫ 输出是否符合命令语义？

⚫ 默认行为是否符合预期？

⚫ 是否存在与常见实现不一致的情况？

# B.2 选项解析风险（Option Parsing Risk）

指命令行选项解析过程中的潜在问题。

思考方向：

⚫ 多个选项组合时是否存在歧义？

⚫ 选项顺序是否影响结果？

⚫ 非法或未知选项如何处理？

⚫ 选项与普通参数混合时是否正确区分？

# B.3 输入语义解析风险（Input Semantic InterpretationRisk）

指程序对输入内容（而非仅选项）进行解释时可能产生的语义偏差。

思考方向：

⚫ 输入数据是否被正确解析？

⚫ 不同输入模式是否触发不同语义路径？

⚫ 是否存在隐含规则？

⚫ 是否存在解析歧义或不一致行为？

# B.4 边界输入风险（Boundary Input Risk）

指极端或边界输入条件可能引发异常行为。

思考方向：

⚫ 空输入或无参数情况如何处理？

⚫ 超长输入是否稳定？

⚫ 参数数量极少或极多时是否异常？

⚫ 是否存在隐含边界条件？

# B.5 错误处理风险（Error Handling Risk）

指程序在异常或非法输入情况下的行为风险。

思考方向：

⚫ 是否存在未处理异常路径？

⚫ 是否静默失败？

⚫ 是否给出错误提示？

⚫ 是否返回合适的状态码？

# B.6 脚本控制风险（Script Control Risk）

指该组件在脚本或自动化环境中使用时可能带来的系统级影响。

思考方向：

⚫ 输出格式是否可能影响脚本逻辑判断？

⚫ 换行行为是否可能改变控制流程？

⚫ 输出内容是否可能被误解析？

. 输出差异是否可能在管道或组合命令中被放大？

# B.7 IO 行为风险（IO Behavior Risk）

指与输出行为相关的系统层面风险。

思考方向：

⚫ 输出是否稳定？

⚫ 是否存在缓冲或刷新问题？

⚫ 在重定向或管道环境下是否正常？

. 是否正确使用标准输出或错误输出？

# 使用要求

在风险识别阶段：

1. 每条风险必须明确属于某一维度（可跨维度）

2. 风险描述必须具体化（不可只写“存在边界问题”）

3. 必须说明风险后果及影响范围

若识别出未包含在上述维度中的合理风险类别，可在报告中说明并酌情加分。

# D.1 系统建模

# D.1.1 功能语义识别

echo 用于将输入参数输出到标准输出。

默认情况下，所有参数按顺序输出，并在末尾添加换行。

选项 -n 用于取消末尾换行。

选项 -e 用于启用反斜杠转义字符解析。

选项 -E 用于显式关闭转义解析，默认情况下通常等价于-E。

当启用转义解析时：

\n 表示换行（结构变化行为）

⚫ \t 表示制表符（字符替换行为）

⚫ \c 表示终止输出，并忽略后续字符，同时不再输出换行（控制行为）

多个参数之间默认以空格分隔输出。

# D.1.2 输入域识别

# 输入结构

1 echo [option] [string]... 

# 输入构成

# 1 选项集合

1 -n 取消换行

2 -e 启用反斜杠转义解析

3 -E 显式禁用转义解析（默认行为）

# 2 普通参数

1 字符串

2 可能包含转义字符

3 可能为空

# 3 输入组合

1 echo hello 

2 echo -n hello 

3 echo -e "a\nb" 

4 echo -n -e "hello" 

5 echo "a" "b" 

# D.1.3 关键判定逻辑识别

判定逻辑1：是否存在 -n

影响：决定输出末尾是否添加换行。

判定逻辑2：是否启用 -e

影响：决定是否对字符串中的反斜杠序列进行解析。

判定逻辑 3：是否出现 \c

影响：终止后续输出并取消换行。

判定逻辑4：是否存在多个参数

影响：多个参数之间是否插入空格。

2 转义字符类型

1 是否启用转义

# D.3 测试决策

在实际项目中，测试资源通常有限，无法为每一个风险单独设计测试。因此需要根据风险排序选择有限数量的测试用例。

本示例将测试数量限制为 5 个。

设计原则：

1 优先覆盖高风险项

2 尽量通过一个测试覆盖多个风险，避免为同一风险设计多个专用测试

3 放弃低风险或重复测试

# D.3.1 测试资源分配过程

风险排序用于确定优先验证的风险区域。

在测试资源有限的情况下，测试设计通常会优先覆盖高风险行为，并尽量通过一个测试同时触达多个风险。因此测试选择不一定严格按照风险排序逐项进行，而是结合程序结构和风险分布进行综合决策。

在本示例中，转义解析属于核心功能机制，因此相关测试（T1–T3）优先于输入结构测试（T4、T5）。

# D.5.1 本轮决策有效性

# （1）风险排序是否合理

本示例通过系统建模识别了 8 条潜在风险，并根据严重性与发生概率进行排序。排序结果中，最高风险集中在以下行为路径：

• 输出控制（R6：\c 终止逻辑）

• 选项组合（R3：-n 与 -e 组合）

• 转义解析（R4：\n 解析）

这些风险直接影响程序输出行为，并可能影响脚本执行结果，因此被认为是需要优先验证的关键风险。

# （2）测试资源是否集中在关键风险

在测试数量限制为 5 个的条件下，测试设计优先覆盖上述高风险路径：

• T1 覆盖 R6、R2（输出终止逻辑）

• T2 覆盖 R3、R4、R1、R2（选项组合与转义解析）

• T3 补充转义解析路径（R5、R2）

同时，测试集合还覆盖了参数结构相关风险：

• T4 覆盖多参数输出风险（R7）

• T5 覆盖无参数边界输入（R8）

总体上：共设计 5 个测试，触达 8 个风险

由于多个风险来源于相同输入机制，一个测试输入可以同时触达多个风险，因此无需为每个风险分别设计独立测试用例。

# （3）是否存在覆盖率较高但风险未控制

覆盖分析结果显示：语句覆盖率： $9 0 . 2 \%$ ；判定覆盖率： $7 6 . 5 \%$ 

未覆盖路径主要包括：

• 无效选项处理

• 八进制转义解析

• 错误处理路径

这些路径对应的行为在当前风险排序中优先级较低，且多数需要特殊输入或环境才能触发，因此未纳入主要测试集合。

该结果表明：测试设计优先控制关键行为风险，而不是单纯追求更高的代码覆盖率。

# 若用于嵌入式启动脚本：

⚫ 增加输入解析相关测试

⚫ 增加边界输入验证

⚫ 减少低风险行为测试

# E.2 运行时风险示例

内存管理工具在软件开发中主要用于检测和调试内存使用问题，例如内存泄漏、未初始化内存访问、双重释放、越界访问等。在某些情况下，内存管理相关的 Bug 并不会影响程序的运行结果，因此仅通过程序的输出并不能发现这些问题。这种情况下，我们需要借助内存管理工具来对程序的内存使用进行分析和调试。

Valgrind 是一款常见的内存管理工具，用于检测内存泄漏和错误的内存操作（如未初始化内存的使用、无效的内存访问等）。以下以 echo.c 中内存管理相关代码为例，在代码中注入越界访问和内存泄漏缺陷并借助 Valgrind 发现问题。

# E.2.1 越界访问

```txt
1 buflen = 0;  
2 pp = argv;  
3 while ((arg = *pp) != NULL) {  
4     buflen += strlen(arg); // 少了 "+"1"  
5     pp++;  
6 }  
7 out = buffer = xmalloc(buflen); // 少了 "+"1"
```

# 问题分析：

1. 每个字符串长度通过 strlen(arg) 累加到 buflen，但忽略了字符串末尾的空字节（通常需要为结尾的 \0 预留空间）。

2. 最后调用 xmalloc(buflen) 分配内存，但分配空间不足（缺少 $+ 1$ ），后续操作可能会导致缓冲区溢出。

以./busybox echo "Hello" "World"为例，本应分配 13 字节的空间，但是实际只分配了10 字节，只包括了字符串的总长度。

# E.2.2 内存泄露

```c
1 do_write:  
2 /* Careful to error out on partial writes too (think ENOSPC!) */  
3 errno = 0;  
4 err = full_write(STDOUT_FILENO, buffer, out - buffer) != out - buffer;  
5 if (err) {  
6 bbsimple_perror msg(bb msg_write_error);  
7 }  
8 // free(buffer); // 错误：内存未释放，导致泄漏  
9 return err;
```

# 问题分析：

1. buffer 是通过 xmalloc() 动态分配内存的，为避免内存占用，程序返回前应释放这部分内存。

2. 未调用 free(buffer)会导致内存泄漏，尤其当程序执行多次或作为长期运行的应用时，内存泄漏会逐渐积累，导致资源耗尽。

# 越界写入

⚫ Valgrind 检测到程序尝试写入超出分配内存范围的地址。

 分配内存的地址：0x4a4e040，大小为 10 字节。

越界写入的地址：

第一次：0x4a4e04a（第 11 字节）

第二次：0x4a4e04b（第 12 字节）

#  调用栈：

分配内存的函数：malloc（由 0x116C8F 调用） 

越界写入的函数：stpcpy 和其他未知函数（0x1160AC 和 0x1160C7）。

# ⚫ Syscall param write(buf) points to unaddressable byte(s)

系统调用 write 的参数指向了未分配的内存区域（即越界的内存）。

 这表明程序尝试将越界写入的内容传递给 write 系统调用，导致潜在的错误。

 问题地址：0x4a4e04a，这是越界写入的第 11 字节。

# 内存泄漏

 程序分配了 10 字节的内存，但在程序退出时没有释放。

 Valgrind 报告了 1 个内存泄漏：

泄漏的内存块：10 字节。

分配位置：malloc（由 0x116C8F 调用） 

# E.3 单元测试

单元测试（Unit Testing）是一种软件测试的方法，用于验证软件中的最小可测试单元（通常是函数或模块）是否能够按预期工作。其主要特点包括：

⚫ 目标明确：针对函数的输入和输出进行验证，确保实现功能正确。

⚫ 局部性强：只关注特定模块或函数的功能，而不会验证整个系统。

⚫ 自动化：通常使用专用框架（如 Check、Google Test）实现测试自动化。

Check 是一个轻量级的 ${ \mathsf { C } } / { \mathsf { C } } + +$ 单元测试框架，用于创建测试套件和测试用例。它支持编写简单的断言语句，并提供便捷的测试报告工具。

Check 的测试程序由以下部分组成：

1. 测试用例（Test Case）：定义具体的测试内容，通过断言验证功能。

2. 测试集（Test Case Collection / TCase）：将功能相关的测试用例归类为一个集合。

3. 测试套件（Test Suite）：多个测试用例集的高级集合，通常表示一个模块或项目。

4. 测试运行器（Test Runner / SRunner）：执行整个测试套件并生成测试报告。

本实践的测试对象为完整的命令行程序，测试通过执行程序并比较标准输出结果来验证其行为，因此测试层级更接近于程序级或系统级测试。

check 是一种面向 C 语言函数或模块的单元测试框架，通常用于在代码内部调用被测函数并通过断言验证返回值或内部状态。在本实验场景下，如果使用 check 实现测试，需要额外封装程序入口或重构代码结构，反而增加实验复杂度。因此本手册的主要实验流程未将 check 作为核心工具，而是将其使用方法放在附录中作为扩展内容，并展示了如何用 check 实现与实验测试用例等价的自动化测试。

# E.4 变异测试验证测试有效性

Mull 是一个针对 C 和 ${ \mathsf { C } } + +$ 的实用动态突变测试工具，它基于强大的 LLVM 框架运行。通过引入程序代码中的“突变”，Mull 帮助开发者检测测试用例是否能够有效地覆盖所有可能的执行路径，提高软件质量与稳定性。

Mull的核心是利用 LLVM 编译器基础设施，它可以修改（即“突变”）源代码的中间表示(IR)，生成一系列变异体。这些变异体是原程序的微小变化版本，如果测试套件不能使它们“死亡”（即导致突变体出错），那么就说明测试用例可能不够全面。

# 1. 当使用 -n 选项时：

 无论 nflag 最初是 1 还是 42，代码都会在处理 -n 选项时将 nflag 强制设为0。

因此，后续 if (nflag) 的判断结果都为假，都不打印换行符。

原始代码和变异代码在这种情况下行为完全一致。

# 2. 当不使用 -n 选项时：

原始代码：nflag 保持为 1。if (nflag) 为真，打印换行符。

 变异代码：nflag 保持为 42。if (nflag)也为真，打印换行符。

原始代码和变异代码在这种情况下行为也完全一致，因为 1 和 42 在布尔判断中都代表“真”。

因此，变异 charnflag=42;在上下文中就是一个 等效变异体 。无论 nflag 是 1 还是 42（前提是没有 -n 选项将其重置为 0），if (nflag) 条件的求值结果都将为真，因为 1 和42 都是非零值。程序的输出结果是保持不变的。
