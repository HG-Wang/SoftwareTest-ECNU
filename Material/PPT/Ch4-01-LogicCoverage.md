# Ch4.1 逻辑覆盖标准（Semantic Logic Coverage Criteria）

## 课程目标

1. 识别逻辑表达式中的谓词（predicate）和子句（clause）
2. 比较 PC、CC、CoC，解释它们的局限性
3. 解释 ACC 中每个子句如何独立影响谓词（GACC、CACC、RACC）
4. 分析不同逻辑覆盖标准下逻辑表达式的测试需求

---

## 1. 动机

考虑以下逻辑表达式：

```
if ((a < b) || (f(z) && D && (m >= n*o))) { ... }
```

- 2 个测试用例足够测试该逻辑表达式吗？
- 如果已实现分支覆盖，是否充分？

**问题：** 传统的语句覆盖和分支覆盖将复杂的逻辑表达式视为**单一的布尔结果**，没有显式测试表达式内部的逻辑语义。

**解决方案：** 逻辑覆盖 — 一组专门测试逻辑表达式的标准，关注表达式的不同部分如何影响整体结果。

---

## 2. 基本概念

### 2.1 谓词与子句

| 术语 | 英文 | 定义 | 示例 |
|------|------|------|------|
| **谓词** | Predicate | 求值为布尔值的逻辑表达式 | `(a < b) \|\| (f(z) && D)` |
| **子句** | Clause | 不含逻辑运算符的逻辑表达式（基本单元） | `a < b`、`f(z)`、`D` |

- 谓词可包含：布尔变量、比较运算符（>、<、== 等）、布尔函数调用
- 子句通过逻辑运算符连接：¬（非）、∧（与）、∨（或）、→（蕴含）、⊕（异或）、↔（等价）
- 在编程中，谓词常称为**判定（Decision）**，子句常称为**条件（Condition）**

### 2.2 子句统计

对 63 个开源程序的 400,000+ 个谓词分析：

| 子句数 | 比例 |
|--------|------|
| 1 个子句 | 88.5% |
| 2 个子句 | 9.5% |
| 3 个子句 | 1.35% |
| 4+ 个子句 | 0.65% |

大多数谓词非常简单，但控制软件中可能有少量包含多个子句的复杂谓词。

## 3. 基础覆盖标准

### 3.1 谓词覆盖 PC (Predicate Coverage)

> **定义：** 谓词 p 必须至少求值为 **true** 一次，至少求值为 **false** 一次。

示例：`p = ((a < b) ∨ D) ∧ (m >= n*o)`

| 测试 | a | b | D | m | n | o | p |
|------|---|---|---|---|---|---|---|
| true | 5 | 10 | T | 1 | 1 | 1 | T |
| false | 10 | 5 | F | 1 | 1 | 1 | F |

**局限性：** PC 不保证所有子句都取到 true 和 false。例如 `P = A ∧ B`，若测试为 (A=T, B=T) 和 (A=F, B=T)，则 B 始终为 true。

### 3.2 子句覆盖 CC (Clause Coverage)

> **定义：** 每个子句必须至少求值为 **true** 一次，至少求值为 **false** 一次。

**局限性：** CC 不保证谓词取到 true 和 false。例如 `p = A ∨ B`，测试 (A=T, B=F) 和 (A=F, B=T)，每个子句都取了 true 和 false，但谓词始终为 true。

### 3.3 组合覆盖 CoC (Combinatorial Coverage)

> **定义：** 测试子句值的**所有可能组合**。也称多重条件覆盖 (Multiple Condition Coverage)。

- 对 N 个子句，需要 **2^N** 个测试
- 简单、全面，但成本随子句数指数增长
- 对 3 个以上子句的谓词通常不现实

---

## 4. 有效子句覆盖 ACC (Active Clause Coverage)

### 4.1 核心思想

子句覆盖的弱点：子句值不总能影响谓词。要真正测试一个子句的结果，该子句应是谓词值的**决定因素**。

> **决定（Determination）：** 谓词 p 中的子句 cᵢ 称为**主子句（major clause）**，当且仅当剩余的**次子句（minor clauses）** cⱼ 的值使得改变 cᵢ 能改变 p 的值时，cᵢ 决定 p。

示例：`P = A ∨ B`
- 若 B = true，则 A **不**决定 P（A 变化，P 不变）
- 若 B = false，则 A **决定** P（A = true → P = true；A = false → P = false）

### 4.2 ACC 的三种变体

> **Active Clause Coverage (ACC)：** 对每个子句 cᵢ：cᵢ = true 一次，cᵢ = false 一次。每次测试中，cᵢ（主子句）必须决定 p。

ACC 对次子句的要求存在歧义，由此产生三种变体：

#### GACC — 一般有效子句覆盖

> 次子句**不需要**相同。cᵢ 决定 p 即可。

- **最弱**的 ACC 变体
- **不保证**谓词覆盖
- 示例：`p = a ⊕ b`，测试 (a=T, b=F → p=T) 和 (a=F, b=T → p=T) → a 决定 p，但 p 始终为 true

#### RACC — 受限有效子句覆盖

> 次子句**必须相同**。且 p 必须变化。

- **最强**的 ACC 变体
- 与 **MC/DC** 密切相关，广泛用于安全关键软件（航空等）
- 清晰的因果关系，但可能导致不可行的测试需求
- 示例：`p = a ⊕ b`，测试 (a=T, b=F → p=T) 和 (a=F, b=F → p=F) → 仅 a 变化，效果可观察

#### CACC — 关联有效子句覆盖

> 次子句**可以不同**，但 p **必须变化**。

- **中等**强度
- 保证谓词覆盖
- 示例：`p = a ⊕ b`，测试 (a=T, b=F → p=T) 和 (a=F, b=F → p=F) → a 的效果在 p 中可观察

### 4.3 ACC 变体对比

| 标准 | 次子句固定？ | 保证 p 变化？ | 强度 |
|------|------------|-------------|------|
| GACC | 否 | 否 | 最弱 |
| CACC | 部分 | 是 | 中等 |
| RACC | 是 | 是 | 最强 |

### 4.4 示例：`p = a ∧ (b ∨ c)` 的 CACC 和 RACC

真值表：

| 行 | a | b | c | p |
|----|---|---|---|---|
| 1 | T | T | T | T |
| 2 | T | T | F | T |
| 3 | T | F | T | T |
| 4 | T | F | F | F |
| 5 | F | T | T | F |
| 6 | F | T | F | F |
| 7 | F | F | T | F |
| 8 | F | F | F | F |

**CACC：** a 决定 p，p 必须变化，(b, c) 可以不同
- 选择行 1 和 5（或 2 和 6，或 3 和 7）

**RACC：** a 决定 p，p 必须变化，(b, c) 必须相同
- 选择 (1, 5)、(2, 6)、(3, 7) 各一对

## 5. 非活动子句覆盖 ICC (Inactive Clause Coverage)

ACC 确保"主子句"影响谓词；ICC 采取相反方法 — 主子句**不影响**谓词。

> **ICC 定义：** 对每个主子句 cᵢ，选择次子句使 cᵢ **不**决定 p。TR 包含四个需求：(1) cᵢ=T 且 p=T，(2) cᵢ=F 且 p=T，(3) cᵢ=T 且 p=F，(4) cᵢ=F 且 p=F。

- **GICC：** 次子句值不需要相同
- **RICC：** 次子句值必须相同
- ICC 系列标准**包含 PC**（对所有子句考察 p=T 和 p=F 场景）

---

## 6. 不可行性与包含关系

### 6.1 不可行测试需求

示例：`(a > b ∧ b > c) ∨ c > a`

- `(a > b) = true, (b > c) = true, (c > a) = true` 是不可行的（矛盾：a > b > c > a）

识别不可行测试需求是困难的，且通常是**不可判定**的。软件测试是工程而非科学。

### 6.2 包含关系

```
CoC ⊇ RACC ⊇ CACC ⊇ GACC
CoC ⊇ RACC ⊇ RICC
CoC ⊇ RACC ⊇ GICC
ICC 系列 ⊇ PC
```

---

## 7. 使子句决定谓词的方法

### 7.1 定义法

- `p_{c=true}`：将 p 中所有 c 替换为 true
- `p_{c=false}`：将 p 中所有 c 替换为 false
- `p_c = p_{c=true} ⊕ p_{c=false}`（异或连接）

求解后，`p_c` 描述了使 c 决定 p 所需的次子句值。

### 7.2 示例

| 谓词 | p_a | 含义 |
|------|-----|------|
| `p = a ∨ b` | `¬b` | b 必须为 false，a 才能决定 p |
| `p = a ∧ b` | `b` | b 必须为 true，a 才能决定 p |
| `p = a ∨ (b ∧ c)` | `¬b ∨ ¬c` | b 或 c 中至少一个为 false |

### 7.3 XOR 恒等式

```
A ⊕ (A ∧ b) = A ∧ ¬b
A ⊕ (A ∨ b) = ¬A ∧ b
```

---

## 8. 测试数量

- MC/DC 和 RACC 至少需要 **n + 1** 个测试，但始终少于 **2n** 个测试
- 当 n < 4 时，n + 1 个测试通常足够
- 随着 n 增大，所需测试数趋近 2n

---

## 9. 实践指导

| 子句数 | 推荐标准 |
|--------|---------|
| 1 个子句 | PC 即可 |
| 2-3 个子句 | CoC 可行 |
| 多个子句 | ACC / MC/DC |

逻辑覆盖不是关于更多测试，而是关于测试**有意义的行为** — 每个条件是否真正影响结果。

---

## 10. 术语对照

| 英文 | 中文 | 说明 |
|------|------|------|
| Decision Coverage / Branch Coverage | 判定覆盖 / 分支覆盖 (DC) | 判定的 true 和 false 至少各执行一次 |
| Condition Coverage | 条件覆盖 (CC) | 每个条件的 true 和 false 至少各执行一次 |
| Predicate Coverage | 谓词覆盖 (PC) | 每个谓词 P 的 true 和 false 各执行一次 |
| Clause Coverage | 子句覆盖 (CC) | 每个子句 C 的 true 和 false 各执行一次 |
| MC/DC | 修正条件判定覆盖 | 每个条件独立影响判定 true 和 false 各一次 |
| C/DC | 条件判定覆盖 | 同时满足 DC 和 CC |
