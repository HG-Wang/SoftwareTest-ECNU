Task 1：Min() 函数变体分析

任务： 为以下 Min() 函数的变体 Δ2、Δ4、Δ5、Δ6 提供到达条件、感染条件、传播条件和杀死变体的测试用例。

原始程序 
int Min(int A, int B) {
    int minVal;
    minVal = A;
    if (B < A) {
        minVal = B;
    }
    return (minVal);
}
嵌入变体 
int Min(int A, int B) {
    int minVal;
    minVal = A;
    Δ1 minVal = B;
    if (B < A)
    Δ2 if (B > A)
    Δ3 if (B < minVal)
    {
        minVal = B;
        Δ4 Bomb();
        Δ5 minVal = A;
        Δ6 minVal = failOnZero(B);
    }
    return (minVal);
}

程序控制流

// 原程序控制流
[1] minVal = A
[2] if (B < A)
      ├── true  → [3] minVal = B
      └── false → (跳过)
[4] return (minVal)
Control Flow

变体 Δ2：条件 B < A → B > A

变异点在 if 条件处，将 B < A 反转为 B > A。

RIPR 条件分析过程逻辑表达式
Reachability 条件判断语句在每次函数调用时都会被执行。执行路径无条件经过该语句。 无条件（始终到达）
Infection 原程序在 B < A 为 true 时进入 if 分支；变体在 B > A 为 true 时进入。要使分支走向不同，需要 (B < A) ≠ (B > A)，即 A ≠ B。 A ≠ B
Propagation 情况 1：B < A — 原程序进入分支 (minVal=B)，变体不进入 (minVal=A)。输出 B vs A，不同。
情况 2：B > A — 原程序不进入 (minVal=A)，变体进入 (minVal=B)。输出 A vs B，不同。
两种情况下输出均不同。传播条件与感染条件相同。 		A ≠ B

杀死测试：

AB原程序输出变体输出结果
533 (minVal=B)5 (minVal=A)✅ 杀死
272 (minVal=A)7 (minVal=B)✅ 杀死

变体 Δ4：minVal = B → Bomb()

变异点在 if 分支体内，将 minVal = B 替换为 Bomb()（抛出异常）。

RIPR 条件分析过程逻辑表达式
Reachability 必须进入 if 分支体才能执行到 Bomb()。需要条件 B < A 为 true。 B < A
Infection Bomb() 抛出异常，程序状态完全不同于正常执行 minVal = B。进入分支即感染。 B < A（进入即感染）
Propagation 异常会直接传播到函数外部，程序终止并抛出异常，而非返回正常值。异常自动传播，无需额外条件。 B < A（异常自动传播）

杀死测试：

AB原程序输出变体输出结果
533异常 (Bomb)✅ 杀死
1011异常 (Bomb)✅ 杀死

变体 Δ5：minVal = B → minVal = A

变异点在 if 分支体内，将 minVal = B 替换为 minVal = A。

RIPR 条件分析过程逻辑表达式
Reachability 必须进入 if 分支体。需要条件 B < A 为 true。 B < A
Infection 原程序执行 minVal = B，变体执行 minVal = A。进入分支条件已要求 B < A，因此 B ≠ A，状态不同。 B < A（蕴含 A ≠ B）
Propagation return 语句返回 minVal。原程序返回 B，变体返回 A。由于 B < A 即 B ≠ A，输出不同。 B < A（蕴含输出不同）

杀死测试：

AB原程序输出变体输出结果
533 (minVal=B)5 (minVal=A)✅ 杀死
1011 (minVal=B)10 (minVal=A)✅ 杀死

变体 Δ6：minVal = B → minVal = failOnZero(B)

变异点在 if 分支体内，将 minVal = B 替换为 minVal = failOnZero(B)。

failOnZero(B) 的行为： 当 B ≠ 0 时返回 B 本身；当 B = 0 时抛出异常（或返回不同于 0 的值）。
RIPR 条件分析过程逻辑表达式
Reachability 必须进入 if 分支体。需要条件 B < A 为 true。 B < A
Infection 若 B ≠ 0： failOnZero(B) = B，minVal 值相同，无感染。
若 B = 0： failOnZero(0) 抛出异常，程序状态完全不同。 		B = 0（且 B < A → A > 0）
Propagation 当 B = 0 时，failOnZero(0) 抛出异常，直接传播到输出。无需额外条件。 B = 0（异常自动传播）

杀死测试：

AB原程序输出变体输出结果
100异常 (failOnZero)✅ 杀死
500异常 (failOnZero)✅ 杀死

Task 1 总结

变体变异描述ReachabilityInfectionPropagation测试用例
Δ2 条件 B<A → B>A 无条件 A ≠ B A ≠ B (5, 3)
Δ4 minVal=B → Bomb() B < A B < A B < A (5, 3)
Δ5 minVal=B → minVal=A B < A B < A B < A (5, 3)
Δ6 minVal=B → failOnZero(B) B < A B = 0 B = 0 (1, 0)

Task 2：findVal() 方法变体分析

任务： 对于 findVal() 方法中的变体（i = 0 → i = 0+1，即 i = 1），回答以下四个问题。

原始程序

// Effects: If numbers null throw NullPointerException
//    else return LAST occurrence of val in numbers[]
//    If val not in numbers[] return -1
public static int findVal(int numbers[], int val) {
    int findVal = -1;
    for (int i = 0; i < numbers.length; i++)
        if (numbers[i] == val)
            findVal = i;
    return (findVal);
}
Java

变体描述

原始 
for (int i = 0; ...
变体 
for (int i = 0 + 1; ...  // 即 i = 1

变体将循环起始值从 i = 0 改为 i = 1，导致 numbers[0] 永远不会被检查。

分析：

  • 变体位于 for 循环的初始化语句 int i = 0 → int i = 1
  • 在 Java 中，for 循环的初始化部分在进入循环时无条件执行
  • 函数被调用后，执行流必然经过 for 循环头
  • 即使 numbers = null，NPE 发生在 numbers.length（条件检查阶段），而初始化 i = 0/1 已经执行完毕
结论：不可能。 无论什么测试输入，for 循环初始化语句始终会被执行，变体始终被到达。不存在使变体不被到达的测试输入。

分析：

  • 到达变体后，原程序中 i = 0，变体中 i = 1
  • 此时变量 i 的值已经不同 — 这就是程序状态的差异
  • 根据 Infection 的定义（变异语句执行后程序状态不同），只要执行到该语句，状态就一定不同
结论：不可能。 i = 0 和 i = 1 是不同的值，到达即感染。不存在使到达但不感染的测试输入。

分析：

  • 感染已满足（i 值不同：0 vs 1）
  • 要使差异不传播到输出，需要变体和原程序的返回值相同
  • 变体跳过了 numbers[0] 的检查
  • 如果 numbers[0] ≠ val，则即使原程序检查了 numbers[0]，也不会设置 findVal = 0
  • 此时跳过 numbers[0] 不影响最终结果，差异不传播

测试用例： numbers = [1, 2, 3], val = 2

步骤原程序 (i=0 起)变体 (i=1 起)
i=0numbers[0]=1 ≠ 2, 跳过(不执行)
i=1numbers[1]=2 == 2, findVal=1numbers[1]=2 == 2, findVal=1
i=2numbers[2]=3 ≠ 2, 跳过numbers[2]=3 ≠ 2, 跳过
返回11
结论：可以。 numbers = [1, 2, 3], val = 2 满足感染但不传播（输出相同，均为 1）。

分析：

  • 要杀死变体，需要变体的输出与原程序不同
  • 变体跳过了 numbers[0] 的检查
  • 需要 numbers[0] == val 且 val 在数组中最后一次出现在 index 0
  • 即 val 不出现在 index 1 及之后的位置

测试用例 1： numbers = [5], val = 5

步骤原程序变体
i=0numbers[0]=5 == 5, findVal=0(不执行)
循环结束返回 0返回 -1

输出不同（0 vs -1）→ 变体被杀死 ✅

测试用例 2： numbers = [5, 3, 7], val = 5

步骤原程序变体
i=0numbers[0]=5 == 5, findVal=0(不执行)
i=1numbers[1]=3 ≠ 5numbers[1]=3 ≠ 5
i=2numbers[2]=7 ≠ 5numbers[2]=7 ≠ 5
返回0-1

输出不同（0 vs -1）→ 变体被杀死 ✅

结论：可以。 numbers = [5], val = 5 或 numbers = [5, 3, 7], val = 5 均可杀死变体。核心条件是 val 仅出现在数组的 index 0 位置。

Task 2 总结

问题是否可能测试用例说明
(a) 不到达 ❌ 不可能 — for 循环初始化无条件执行
(b) 到达不感染 ❌ 不可能 — i=0 vs i=1 状态始终不同
(c) 感染不传播 ✅ 可能 [1,2,3], val=2 numbers[0]≠val，跳过不影响结果
(d) 杀死变体 ✅ 可能 [5], val=5 val 仅出现在 index 0，变体漏检
解题关键： 在分析变体时，按照 R → I → P 的顺序逐步分析。首先确定执行路径是否经过变异点（R），然后分析变异是否导致状态差异（I），最后追踪差异是否传播到输出（P）。对于程序级变异，重点关注变量值的变化和控制流的差异。
