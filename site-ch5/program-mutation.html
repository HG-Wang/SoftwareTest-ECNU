经典案例：Min() 函数

以下是一个经典的变异测试案例，展示了如何对一个简单的求最小值函数进行变异分析：

原始程序

int Min(int A, int B) {
    int minVal;
    minVal = A;
    if (B < A) {
        minVal = B;
    }
    return (minVal);
}
C++

嵌入变体的程序

在原始程序的基础上，通过变异运算符生成了 6 个变体：

int Min(int A, int B) {
    int minVal;
    minVal = A;
    Δ1  minVal = B;              // 变体1：额外赋值
    if (B < A)
    Δ2  if (B > A)               // 变体2：条件反转
    Δ3  if (B < minVal)          // 变体3：替换条件
    {
        minVal = B;
        Δ4  Bomb();               // 变体4：炸弹语句
        Δ5  minVal = A;           // 变体5：赋值替换
        Δ6  minVal = failOnZero(B); // 变体6：函数替换
    }
    return (minVal);
}

等价变体 (Equivalent Mutant)

等价变体是变异测试中最棘手的问题之一。它是指与原程序行为完全相同的变体 — 不存在任何测试能杀死它。

案例：变体 Δ3 的等价性证明

原程序 
minVal = A;
if (B < A)    // 原始条件
{
    minVal = B;
}
变体 Δ3 
minVal = A;
if (B < minVal)  // 修改后的条件
{
    minVal = B;
}

证明过程：

  1. 在 if 语句之前，执行了 minVal = A
  2. 因此在条件判断处，minVal = A
  3. 变体的条件是 B < minVal，代入得 B < A
  4. 这与原程序的条件 B < A 完全相同！
  5. Infection 条件：(B < A) ≠ (B < minVal) → (B < A) ≠ (B < A) → 矛盾
结论： 变体 Δ3 是等价变体。无论使用什么测试输入，变体 Δ3 的行为都与原程序完全相同。等价变体的存在是变异测试的主要挑战之一，因为判断一个变体是否等价通常是不可判定的问题。

弱变异案例分析

案例：变体 Δ1 的弱变异分析

变体 Δ1 在 if 之前插入了 minVal = B：

原程序 
minVal = A;
// (无额外语句)
if (B < A) {
    minVal = B;
}
return minVal;
变体 Δ1 
minVal = A;
minVal = B;  // ← 插入
if (B < A) {
    minVal = B;
}
return minVal;

RIP 分析：

条件分析表达式
R 插入的语句在每次调用时都会执行 无条件
I 原程序 minVal=A，变体 minVal=B → 需要 A≠B A ≠ B
P (强杀死) 需要差异传播到 return。若 B<A：两程序都执行 minVal=B → 输出相同！ B ≥ A（取不同分支）

变异测试为什么有效？

变异测试的基本前提 (Fundamental Premise)：
如果软件包含一个故障，通常会存在一组变体，只有能检测到该故障的测试用例才能杀死这些变体。

这意味着：

  • 如果测试集能杀死所有（非等价的）变体，那么它很可能也能检测到程序中的实际故障
  • 变体引导测试者找到有效的测试集
  • 变异测试被广泛认为是测试的"黄金标准"

✅ 优势

  • 非常强大的测试标准
  • 可以包含其他覆盖标准
  • 有坚实的理论基础
  • 可自动化执行

⚠️ 挑战

  • 计算成本高（变体数量大）
  • 等价变体问题
  • 需要语言特定的变异运算符
  • 执行时间长

变异包含其他标准

通过包含特定的变异运算符，变异测试可以包含其他测试标准：

标准是否被包含说明
节点覆盖 (NC)✅通过变异条件语句包含
边覆盖 (EC)✅通过变异分支条件包含
子句覆盖 (CC)✅通过变异子句包含
一般主动子句覆盖✅对单个测试的需求
关联主动子句覆盖❌对测试对的需求，无法通过弱变异包含
全定义数据流覆盖✅通过变异定义点包含
