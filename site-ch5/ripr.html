RIPR 模型

Reachability, Infection, Propagation, Revealability — 分析测试用例是否能检测到程序故障（或杀死变异体）的四条件分析框架。

RIPR 强杀死 弱杀死 故障检测

模型概述

RIPR 模型来自第 2 章，是分析故障检测条件的系统化框架。在变异测试中，它被用来判断一个测试用例是否能够杀死某个变体。

一个测试用例要成功检测到故障（或杀死变体），必须同时满足以下四个条件：

记忆技巧： RIPR 可以理解为"故障检测的四道关卡" — 故障必须先被到达，然后感染程序状态，接着传播到输出，最后被测试者观察到。任何一道关卡失败，故障都无法被检测到。

四个条件详解

定义： 测试用例必须导致包含故障（或变异）的语句被执行。

如果程序的执行路径不经过故障所在的位置，那么无论故障多么严重，都不可能被检测到。

分析要点：

  • 确定故障/变异点在程序中的位置
  • 分析哪些条件控制着执行路径到达该位置
  • 设计测试输入使程序执行经过该位置

示例： 如果故障在 if (B < A) 的分支体内，则需要 B < A 为真才能到达。

定义： 故障语句（或变异语句）执行后，程序的内部状态必须与正确执行不同。

即使到达了故障位置，如果故障没有导致程序状态发生变化（例如故障代码与正确代码碰巧产生相同结果），那么故障也不会被检测到。

分析要点：

  • 比较原程序和变体在变异点之后的状态
  • 找到使状态不同的输入条件
  • 注意：到达即感染的情况（如 Bomb()）和需要额外条件的情况

示例： 变体 if (B > A) 替换 if (B < A)，需要 A ≠ B 才能使分支判定不同。

定义： 被感染的程序状态必须传播到最终输出，导致输出与正确执行不同。

即使内部状态被感染，如果后续的代码"修复"了这个差异（例如覆盖了被感染的变量），那么故障也不会在输出中体现。

分析要点：

  • 追踪被感染变量在后续代码中的使用
  • 确认差异是否传播到 return 语句或副作用
  • 注意：异常（如 Bomb()）会自动传播

定义： 测试者必须能够观察到输出中的差异。

即使输出不同，如果测试者没有检查相关的输出，或者输出差异被忽略了，故障也不会被发现。

分析要点：

  • 测试用例必须有明确的预期输出
  • 测试者必须比较实际输出和预期输出
  • 在自动化测试中，断言 (assertion) 实现了可观察性

强杀死 vs 弱杀死

根据满足 RIPR 条件的程度，可以定义两种"杀死"变体的方式：

💪 强杀死 (Strongly Killing)

给定程序 P 的变体 m 和测试 t，当且仅当 output(t, P) ≠ output(t, m) 时，称 t 强杀死 m。

满足条件：

  • ✅ Reachability — 到达变异点
  • ✅ Infection — 状态不同
  • ✅ Propagation — 差异传播到输出
  • ✅ Revealability — 输出可观察

这是最严格的杀死方式，要求变体的最终输出与原程序不同。

⚡ 弱杀死 (Weakly Killing)

给定修改位置 l 的变体 m 和测试 t，当且仅当执行到 l 之后，P 的状态 ≠ m 的状态时，称 t 弱杀死 m。

满足条件：

  • ✅ Reachability — 到达变异点
  • ✅ Infection — 状态不同
  • ❌ Propagation — 不要求传播到输出
  • ❌ Revealability — 不要求可观察

弱杀死只要求变异点之后的程序状态不同，不要求传播到最终输出。

强杀死与弱杀死的关系

特性强杀死弱杀死
满足 R + I✅✅
满足 P（传播）✅❌
满足 R（可观察）✅❌
杀死难度更难更容易
分析复杂度更高更低
实际效果更严格通常足够
重要发现： 研究表明，能弱杀死所有变体的测试集通常也能强杀死大部分变体。因此弱变异覆盖 (WMC) 在实践中是一种有效且高效的替代方案。

对应的覆盖标准

覆盖标准定义要求
MC (变异覆盖) 对于每个 m∈M，TR 包含一个需求 — 强杀死 m R + I + P + R
WMC (弱变异覆盖) 对于每个 m∈M，TR 包含一个需求 — 弱杀死 m R + I

RIPR 分析方法

对于给定的程序和变体，分析 RIPR 条件的步骤：

  1. 定位变异点
    找到变体与原程序的差异所在位置。明确变异运算符做了什么修改。
  2. 分析 Reachability
    确定哪些条件控制着执行路径到达变异点。写出到达条件的逻辑表达式。
  3. 分析 Infection
    比较原程序和变体在变异点执行后的状态。确定使状态不同的输入条件。注意：有些变异（如 Bomb()）到达即感染。
  4. 分析 Propagation
    追踪被感染的变量在后续代码中的流动。确认差异是否到达 return 语句或输出。注意异常会自动传播。
  5. 设计测试用例
    根据 R + I + P 条件设计具体的输入值。验证测试用例是否能杀死变体。

案例分析

案例：Min() 函数变体 Δ2

将条件 if (B < A) 修改为 if (B > A)。

原程序 
int Min(int A, int B) {
  int minVal;
  minVal = A;
  if (B < A) {
    minVal = B;
  }
  return minVal;
}
变体 Δ2 
int Min(int A, int B) {
  int minVal;
  minVal = A;
  if (B > A) {  // ← 变异点
    minVal = B;
  }
  return minVal;
}
条件分析表达式
R 条件判断在每次调用时都会执行 无条件（始终到达）
I 原程序在 B<A 时进入分支，变体在 B>A 时进入。需要两个条件的布尔值不同。 A ≠ B
P 若 B<A：原程序 minVal=B，变体 minVal=A → 输出不同。若 B>A：反之亦然。 A ≠ B

杀死测试： A=5, B=3

  • 原程序：B<A 成立 → minVal=3 → 返回 3
  • 变体：B>A 不成立 → minVal=5 → 返回 5
  • 输出不同（3 ≠ 5）→ 变体被杀死 ✅
